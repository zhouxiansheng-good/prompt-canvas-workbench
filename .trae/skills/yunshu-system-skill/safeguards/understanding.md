## 第一层：业务背景挖掘（P-015）

> 在01-init阶段触发，确保AI不只是接受指令，而是主动理解业务背后的"为什么"。

### 业务背景挖掘问卷

当用户提出新任务时，必须主动追问以下维度（至少覆盖3个）：

| 维度 | 追问问题 | 目的 |
|------|----------|------|
| **用户是谁** | 这个功能给谁用？他们的技术水平如何？ | 避免技术方案与用户能力错配 |
| **业务目标** | 做这个功能的业务目标是什么？解决了什么痛点？ | 避免实现"能用但不对"的方案 |
| **成功指标** | 怎么算做成了？有没有可量化的指标？ | 建立验收标准的业务锚点 |
| **现有流程** | 现在是怎么做的？有没有手动流程可以参考？ | 从现有流程中提取业务规则 |
| **边界条件** | 什么情况下不该做？异常场景有哪些？ | 避免遗漏边界处理 |
| **关联系统** | 这个功能和哪些现有系统/模块有交互？ | 建立系统级依赖认知 |
| **数据流向** | 数据从哪来？到哪去？中间怎么变换？ | 理解数据边界和流转逻辑 |
| **非功能需求** | 性能要求？安全要求？合规要求？ | 避免只关注功能忽略质量属性 |

### 业务理解深度评估

完成追问后，评估理解深度：

| 理解深度 | 判定标准 | 处理方式 |
|----------|----------|----------|
| 🟢 深度理解 | 覆盖≥5个维度，能复述业务目标 | 可以进入计划阶段 |
| 🟡 基本理解 | 覆盖3-4个维度，目标清晰但细节模糊 | 补充追问1-2轮后进入计划 |
| 🔴 理解不足 | 覆盖<3个维度，或目标不清晰 | 必须继续追问，不能进入计划 |

### 需求文档自检清单

当用户提供了需求文档时，按以下清单检查文档质量：

| 检查项 | 说明 | 缺失时的处理 |
|--------|------|-------------|
| 业务背景描述 | 文档是否说明了"为什么做" | 追问业务目标 |
| 用户角色定义 | 是否明确了目标用户 | 追问用户画像 |
| 验收标准 | 是否有可验证的完成定义 | 协助补充验收标准 |
| 异常场景 | 是否考虑了错误/边界情况 | 追问异常处理策略 |
| 数据模型 | 是否描述了核心数据实体和关系 | 追问数据流向 |
| 接口约定 | 是否定义了与外部系统的交互 | 追问关联系统 |

## 反模式

- 🚫 只接受用户字面指令不追问业务背景（P-015）
- 🚫 不评估理解深度就进入计划阶段（P-015）
- 🚫 只关注功能忽略非功能需求（P-015边界条件）
