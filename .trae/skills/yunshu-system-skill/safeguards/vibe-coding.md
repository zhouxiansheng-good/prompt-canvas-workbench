# Vibe Coding防护

> 覆盖问题：P-050 90%陷阱 / P-051 技术债务隐蔽积累 / P-052 责任链断裂
> 触发场景：计划阶段（02-plan）、验收阶段（04-accept）
> 加载方式：由02-plan和04-accept组件检测到Vibe Coding场景时触发读取本文件

---

## 问题描述

| 编号 | 问题 | 严重程度 | 具体表现 |
|------|------|---------|---------|
| P-050 | 90%陷阱 | 🟠 严重 | AI能完成前90%，剩下10%最难却搞不定 |
| P-051 | 技术债务隐蔽积累 | 🟠 严重 | 表面合格的代码，逻辑漏洞、假数据常被忽视 |
| P-052 | 责任链断裂 | 🟠 严重 | 开发者对AI生成代码盲目信任，出了问题不知谁负责 |

## 第一层：90%节点预警（P-050）

> 在02-plan阶段触发，提前识别"最后10%"的难点，不让用户意外卡住。

### 90%陷阱识别清单

| 陷阱类型 | 识别信号 | 难度倍增原因 |
|----------|----------|-------------|
| **边界条件** | 正常流程通了，异常流程没处理 | 异常路径数量是正常路径的3-5倍 |
| **性能瓶颈** | 功能正确但性能不达标 | 优化需要理解底层原理，AI不擅长 |
| **集成问题** | 单模块OK，模块间交互出错 | 集成问题需要系统级理解 |
| **用户体验** | 功能有了但体验差 | UX需要人类判断，AI缺乏审美 |
| **安全漏洞** | 功能正确但有安全风险 | 安全需要攻击者思维 |
| **数据一致性** | 单次操作OK，并发/分布式出错 | 需要深入理解数据一致性模型 |
| **遗留兼容** | 新功能OK但破坏了旧功能 | 需要理解全部已有功能 |

### 90%节点预警流程

```
02-plan 任务分解时
  ↓
1. 对每个任务评估"90%陷阱风险"
  ↓
2. 标记高风险任务（涉及上述7类陷阱的任一类）
  ↓
3. 对高风险任务：
   ├─ 拆分"简单90%"和"困难10%"为独立任务
   ├─ 困难10%任务标注⚠️预警标签
   └─ 困难10%任务分配更多时间和验证步骤
  ↓
4. 在计划中明确告知用户：
   "任务X的90%可以快速完成，但最后10%（[具体难点]）可能需要额外时间和人工干预"
```

### 预警标签模板

```markdown
⚠️ 90%陷阱预警：[任务名]
  快速完成部分：[90%的内容描述]
  困难部分：[10%的难点描述]
  陷阱类型：[边界条件/性能/集成/UX/安全/一致性/兼容]
  建议策略：[如何攻克最后10%]
  预估额外时间：[正常时间的X倍]
```

## 第二层：技术债务可见化（P-051）

> 在04-accept阶段触发，让隐蔽的技术债务无处藏身。

### 技术债务检测清单

| 债务类型 | 检测方法 | 危险信号 |
|----------|----------|----------|
| **假数据/硬编码** | 搜索TODO/FIXME/hardcode/魔法数字 | 生产环境用了测试数据 |
| **空实现** | 搜索空函数体/throw new Error("not implemented") | 功能声称完成但实际没实现 |
| **静默吞错** | 搜索空catch/空then/console.log(error) | 错误被吞掉不处理 |
| **测试假阳性** | 检查测试是否只覆盖快乐路径 | 测试全过但Bug频出 |
| **过度简化** | 对比规范和实现，检查是否遗漏复杂场景 | 规范要求10种情况只实现了3种 |
| **临时方案** | 搜索hack/workaround/temp/临时 | 临时方案变成了永久方案 |
| **缺失验证** | 检查输入验证/权限检查/边界处理 | 任何输入都能通过 |

### 技术债务分级

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| 🔴 关键债务 | 会导致生产事故或数据丢失 | 必须立即修复，阻塞交付 |
| 🟠 重要债务 | 会影响可维护性或可靠性 | 应该修复，记录到待办 |
| 🟡 次要债务 | 代码风格/小改进 | 记录，后续迭代处理 |

### 技术债务报告模板

```markdown
### 技术债务报告

| 债务 | 类型 | 级别 | 位置 | 影响 | 修复方案 | 优先级 |
|------|------|------|------|------|----------|--------|
| [描述] | [类型] | 🔴/🟠/🟡 | [文件:行] | [影响范围] | [怎么修] | P0/P1/P2 |
```

## 第三层：责任链明确（P-052）

> 在04-accept阶段触发，确保每个代码变更都有明确的责任人。

### 责任链检查

| 检查项 | 说明 | 缺失时的处理 |
|--------|------|-------------|
| 代码审查签名 | 开发者确认理解并负责该代码 | 必须签名后才能交付 |
| AI生成标记 | 标记哪些代码是AI生成的 | 自动标记AI生成比例 |
| 决策归属 | 每个设计决策有明确归属（人/AI） | 补充决策归属记录 |
| 测试责任 | 每个测试有明确的责任人 | 补充测试责任标记 |
| 回滚责任 | 出问题时谁负责回滚 | 明确回滚责任人 |

### 代码审查签名模板

```markdown
### 代码审查签名

我确认：
1. 我已阅读并理解以上所有变更
2. 我能解释每个关键函数的作用和设计原因
3. 我对代码的正确性负责
4. 我知道出问题时如何回滚

签名：[开发者]
日期：[日期]
AI辅助比例：[X%]
```

### AI生成代码标记规则

| AI参与度 | 标记方式 | 说明 |
|----------|----------|------|
| 100% AI生成 | `// AI-GEN: full` | 完全由AI生成，开发者未修改 |
| AI生成+人工修改 | `// AI-GEN: modified` | AI生成后开发者做了修改 |
| AI辅助+人工主导 | `// AI-GEN: assisted` | 开发者主导，AI辅助补全 |
| 纯人工编写 | 无标记 | 开发者独立编写 |

## 反模式

- 🚫 不识别90%陷阱就开工（P-050意外卡住）
- 🚫 把困难10%和简单90%混在一起（P-050无法预估）
- 🚫 忽视假数据/硬编码/空实现（P-051技术债务隐蔽）
- 🚫 测试只覆盖快乐路径（P-051假阳性）
- 🚫 不标记AI生成代码（P-052责任不清）
- 🚫 "AI写的，不是我负责"（P-052责任链断裂）
- 🚫 临时方案变永久方案（P-051技术债务积累）
- 🚫 不做代码审查签名就交付（P-052责任缺失）
