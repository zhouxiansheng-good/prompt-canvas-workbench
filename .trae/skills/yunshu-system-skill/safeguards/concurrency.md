## 竞态条件识别

| 竞态模式 | 具体表现 | 修复方法 |
|----------|----------|----------|
| 检查后使用(TOCTOU) | 先检查条件再操作，中间被其他线程修改 | 原子操作/加锁 |
| 读写竞争 | 多线程同时读写共享变量 | 互斥锁/原子操作 |
| 死锁 | 两个线程互相等待对方释放锁 | 锁排序/超时机制 |
| 活锁 | 线程不断重试但无法前进 | 退避策略 |
| 资源泄漏 | 异常路径未释放锁/连接 | try-finally/RAII |

## 检查规则

1. **共享状态检查**：是否有多个线程/协程访问同一变量？→ 必须同步
2. **原子性检查**：复合操作是否可能被中断？→ 必须加锁或用原子操作
3. **锁顺序检查**：是否可能按不同顺序获取多个锁？→ 统一锁顺序
4. **异常路径检查**：异常时是否释放了所有锁和资源？→ try-finally
5. **异步错误处理**：Promise/async是否都有catch？→ 必须处理异步错误

## 语言特定检查

| 语言 | 常见陷阱 | 正确写法 |
|------|----------|----------|
| JavaScript | 共享变量无锁 | 使用Atomics或避免共享 |
| Python | GIL误导（IO密集仍有竞态） | threading.Lock/asyncio.Lock |
| Java | synchronized范围过大 | 缩小锁粒度/使用并发集合 |
| Go | goroutine泄漏 | context取消机制 |
| Rust | 编译器保证但逻辑竞态仍可能 | 仍需审查业务逻辑 |

---

## 反模式

- 🚫 共享变量不加锁（竞态条件）
- 🚫 复合操作不保证原子性（检查后使用）
- 🚫 多锁不按固定顺序获取（死锁）
- 🚫 异常路径不释放锁（资源泄漏）
- 🚫 异步操作不处理错误（静默失败）
