# 理解验证门禁

> 覆盖问题：P-053 理解债
> 触发场景：代码验收时（04-accept）
> 加载方式：04-accept门禁触发

---

## 问题描述

| 编号 | 问题 | 严重程度 | 具体表现 |
|------|------|---------|---------|
| P-053 | 理解债 | 🔴 致命 | 你以为代码写完了，其实根本没读懂自己"写"的代码 |

---

## 理解验证流程

```
代码交付前
  ↓
1. 识别关键代码段（核心逻辑/复杂算法/安全相关/边界处理）
  ↓
2. 生成理解验证问题（每个关键段1-2个问题）
  ↓
3. 开发者回答验证问题
  ↓
4. 评估理解深度
  ├─ 通过 → 可以交付
  ├─ 部分通过 → 补充理解后重新验证
  └─ 未通过 → 必须重新阅读代码，理解后重新验证
```

### 理解验证问题模板

| 问题类型 | 问题模板 | 目的 |
|----------|----------|------|
| **为什么这样写** | "这段代码为什么用[X]而不是[Y]？" | 验证是否理解设计决策 |
| **数据流向** | "输入[A]经过这段代码后变成什么？" | 验证是否理解数据变换 |
| **边界情况** | "如果[异常输入]进入这段代码会怎样？" | 验证是否理解边界处理 |
| **副作用** | "这段代码会影响哪些其他模块？" | 验证是否理解系统级影响 |
| **修改影响** | "如果要把[功能]改成[新功能]，需要改哪里？" | 验证是否理解代码结构 |

### 理解深度评估

| 理解深度 | 判定标准 | 处理方式 |
|----------|----------|----------|
| 🟢 深度理解 | 能回答所有类型问题，能预测修改影响 | 可以交付 |
| 🟡 表面理解 | 能回答"是什么"但答不出"为什么" | 补充理解后重新验证 |
| 🔴 理解缺失 | 无法回答关键问题，或回答明显错误 | 必须重新阅读代码 |

### 理解债预警信号

| 信号 | 具体表现 | 严重度 |
|------|----------|--------|
| 跳过阅读 | 直接接受AI输出不阅读 | 🔴 |
| 无法解释 | 被问到代码逻辑时答不上来 | 🔴 |
| 盲目信任 | "AI写的应该没问题" | 🟠 |
| 修改恐惧 | 不敢修改AI生成的代码 | 🟠 |
| 复制粘贴 | 不理解就复制粘贴AI代码 | 🔴 |
