# 元认知自检机制

> 覆盖问题：P-015 业务理解缺失 / P-053 理解债 / P-054 主观偏差 / P-003 长会话退化
> 触发场景：任何输出/行动/决策之前
> 加载方式：各组件关键决策点自动触发

---

## 核心原理

人类专家在行动前会本能地问自己："我真的理解了吗？有没有遗漏？发给对方的信息够不够让他继续行动？"这种**元认知能力**（对自身认知过程的认知）是AI最容易缺失的。

本机制强制AI在执行任何关键动作或发出用户可见输出前，执行一次**递归式自我审视**，从"执行模式"切换到"反思模式"。它不要求展示冗长思考过程，只要求输出前完成结论完整性、证据完整性和缺口处理方式的内部门禁。

研究基础与边界见 `reports/meta-cognition-output-sufficiency-research-20260511.md`。核心判断是：自检/自评/验证问题能提高输出纪律，但不能替代外部证据；因此云舒把自检和调查、验证、引用、提问结合起来。

---

## 自检流程

```
任何输出/行动/决策之前
  ↓
【暂停执行】—— 停止生成内容，进入自检模式
  ↓
【五维信息审视】—— 逐项检查，有缺口的标记为 🟡 或 🔴
  ↓
【缺口评估】
   - 可查证缺口 → 先查证/验证
   - 阻塞缺口 🔴 → 向用户确认，补充信息后再执行
   - 非阻塞缺口 🟡 → 明示假设/限制/后续风险后继续
  ↓
【输出充分性四问】—— 发给用户前检查是否答全、证据足、下一步清楚
```

## 五维信息审视清单

### 维度1：用户意图理解

**自检问题**：
- [ ] 用户的原始需求/目标是什么？
- [ ] 用户的核心痛点是什么？
- [ ] 用户的期望产出是什么？
- [ ] 我是否用自己的话复述了用户意图，并得到确认？

**常见缺口**：
- 把用户的"怎么做"当成了"做什么"（混淆手段和目的）
- 忽略了用户的隐含需求（"我要快"=性能优化+代码简洁）
- 把示例当成了全部需求（用户给了一个例子，但需求是通用的）

**信号**：如果无法用自己的话清晰复述用户意图 → 🔴 理解不完整

---

### 维度2：约束条件识别

**自检问题**：
- [ ] 用户明确说了哪些约束？（时间/技术栈/预算/合规）
- [ ] 用户没说但隐含的约束是什么？（团队能力/现有架构/维护成本）
- [ ] 我是否把所有约束都列出来了？
- [ ] 这些约束之间是否有冲突？

**常见缺口**：
- 只记住了技术约束，忽略了时间约束
- 忽略了"不能破坏现有功能"的隐含约束
- 忽略了"代码必须可维护"的长期约束

**信号**：如果约束清单少于3项 → 🔴 可能遗漏了隐含约束

---

### 维度3：上下文完整性

**自检问题**：
- [ ] 我是否读取了所有相关文件/代码？
- [ ] 我是否理解了代码库的架构和依赖关系？
- [ ] 我是否掌握了项目的技术栈和约定？
- [ ] 我是否遗漏了用户之前提到的关键信息？

**常见缺口**：
- 只读了当前文件，没读调用方/被调用方
- 忽略了项目的README/配置/文档
- 长会话中遗忘了早期的关键决策

**信号**：如果无法画出变更的影响范围 → 🔴 上下文不完整

---

### 维度4：假设验证

**自检问题**：
- [ ] 我做出了哪些假设？（技术选型/业务逻辑/用户意图）
- [ ] 这些假设有证据支持吗？
- [ ] 如果假设错误，后果是什么？
- [ ] 哪些假设需要向用户确认？

**常见缺口**：
- 假设用户用的是最新版本（实际可能是旧版本）
- 假设业务逻辑是A（实际是B）
- 假设"简单修改"不会影响其他模块

**信号**：如果有未经验证的高风险假设 → 🔴 必须确认

---

### 维度5：歧义识别

**自检问题**：
- [ ] 用户的表述中是否有歧义？（"优化一下"/"改得更好"/"尽快"）
- [ ] 我是否把模糊表述当成了明确指令？
- [ ] 哪些词需要用户明确定义？
- [ ] 我是否在没有澄清的情况下做了选择？

**常见缺口**：
- "优化"=性能优化？代码优化？用户体验优化？
- "更好"=更快？更简洁？更健壮？
- "尽快"=今天？本周？不影响其他任务的前提下？

**信号**：如果用户的指令中有形容词但没有量化标准 → 🔴 存在歧义

## 输出充分性四问

> 用户原话抽象为门禁句：**"发出前，先问自己：我是否真的完全掌握，并给出了用户要的全部信息？"**

每次用户可见输出前，必须快速检查：

| 问题 | 通过标准 | 不通过处理 |
|------|----------|------------|
| 1. 是否逐一回答了用户显式问题？ | 每个问题都有直接结论或明确说明无法回答 | 补齐答案；若需用户独有信息，问一个必要问题 |
| 2. 是否给出用户继续行动所需信息？ | 有结论、依据、产物路径/命令/下一步 | 补充路径、命令、验证结果或可执行建议 |
| 3. 关键结论是否有证据？ | 来自本地文件、命令输出、外部来源、验收证据或明确假设 | 能查则查；不能查则标注不确定性 |
| 4. 是否处理了剩余缺口？ | 缺口被分为可查证/阻塞/非阻塞，并有处理动作 | 不许假装完整；按缺口分级处理 |

### 缺口分级

| 等级 | 定义 | 行动 |
|------|------|------|
| 可查证缺口 | 通过读文件、运行命令、联网检索或测试可以补齐 | 先查证，再输出 |
| 🔴 阻塞缺口 | 缺失信息会改变目标、方案、风险或验收判定，且只能由用户提供 | 停下，只问一个必要问题 |
| 🟡 非阻塞缺口 | 不影响当前阶段推进，但影响置信度或后续选择 | 明示假设、限制或后续待办后继续 |

### 输出最小结构

复杂任务的阶段性输出至少包含：

```text
结论：[直接回答]
依据：[文件/命令/来源/推理约束]
缺口：[无 / 非阻塞缺口 / 阻塞问题]
下一步：[可执行动作]
```

简单任务可以压缩成一句话，但仍要内部通过四问。

## 自检执行规则

### 规则1：强制暂停

自检时必须**停止生成任何内容**，先完成五维审视清单，再决定下一步。

**错误示范**：
```
用户：帮我优化这个函数
AI：好的，我先看看...（已经开始执行，没有自检）
```

**正确示范**：
```
用户：帮我优化这个函数
AI：【自检模式启动】
   维度1：用户意图 — "优化"可能有歧义，是指性能优化还是代码简洁度优化？
   维度2：约束 — 用户没有说时间约束和技术约束
   维度3：上下文 — 需要先看这个函数的调用方和使用场景
   维度4：假设 — 我假设用户想要性能优化，但需要确认
   维度5：歧义 — "优化"是歧义词
   → 🔴 有3个缺口，需要向用户确认
AI：在优化之前，我想确认几个问题：
   1. 您说的"优化"是指性能优化还是代码可读性优化？
   2. 这个函数有性能瓶颈吗？瓶颈在哪里？
   3. 优化后需要保持向后兼容吗？
```

### 规则2：阻塞缺口必须填补

任何 🔴 阻塞缺口，必须向用户确认后才能继续执行。可查证缺口必须先调查；🟡 非阻塞缺口可以继续，但必须明示假设、限制或风险。**不允许把阻塞缺口伪装成已掌握信息。**

### 规则3：递归应用

自检不仅在任务开始时执行，在以下节点也必须重新执行：
- 进入新阶段时（01-init → 02-plan → 03-execute）
- 遇到意外情况时（测试失败/代码不符合预期/用户提出新要求）
- 长会话中每完成一个子任务后
- 输出最终答案之前

## 反模式

| 反模式 | 表现 | 后果 |
|--------|------|------|
| 形式化自检 | 快速过一遍清单，全部打勾，没有真正反思 | 自检失效，仍然遗漏信息 |
| 自检后仍猜测 | 发现了缺口但不向用户确认，自己"脑补" | 理解偏差，产出不符合预期 |
| 只在开始时自检 | 执行过程中遇到新信息不再自检 | 中后期偏离目标 |
| 自检过于冗长 | 每个小操作都自检，拖慢节奏 | 用户不耐烦，跳过自检 |

## 与云舒系统的融合

本机制不是独立的，而是嵌入云舒系统各组件的关键决策点：

| 组件 | 嵌入位置 | 自检重点 |
|------|----------|----------|
| 01-init | 任务卡前、向用户提问前 | 意图/约束/DoD 是否足够，问题是否一次只问关键缺口 |
| 02-plan | 计划前、规范前 | 调研证据/假设/风险/决策是否足够支撑任务分解 |
| 03-execute | 声明意图前、状态更新前 | 变更范围/影响/验证证据/下一步是否说清 |
| 04-accept | 验收通过前 | 每条 DoD 是否有新鲜证据，失败项是否完整捕获 |
| 05-deliver | 交付报告前 | 用户离开后能否复跑、回滚、继续维护 |
| 06-recover | 检查点/恢复提示前 | phase_memory、checkpoint、context ledger、下一步是否齐全 |
| 07-subagent | 分派前、接收结果前 | 子任务包是否自包含，子智能体输出是否足够控制器继续判断 |
| 智能体 | 返回控制器前 | 结果、证据、缺口、疑虑是否完整 |
| 全局 | 任何用户可见输出前 | 输出充分性四问 |
