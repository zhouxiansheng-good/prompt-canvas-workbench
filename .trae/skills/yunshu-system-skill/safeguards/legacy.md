## 遗留代码理解策略

### 修改前必做

1. **阅读现有代码**：不要假设，逐行阅读要修改的函数和它的调用链
2. **识别隐式契约**：代码中未明写但实际依赖的行为（如调用顺序、全局状态）
3. **查找测试**：有测试先读测试，测试是最好的文档
4. **查找注释/提交记录**：git log/git blame了解代码变更历史

### 修改策略

| 场景 | 策略 | 风险 |
|------|------|------|
| 小修改（bug修复） | 最小改动原则，不重构 | 低 |
| 中修改（功能扩展） | 先加测试再改代码 | 中 |
| 大修改（架构调整） | 绞杀者模式，逐步替换 | 高 |

### 隐式契约识别清单

| 契约类型 | 识别方法 |
|----------|----------|
| 调用顺序依赖 | 检查是否必须先调A再调B |
| 全局状态依赖 | 检查是否依赖全局变量/单例状态 |
| 隐式类型转换 | 检查是否依赖自动类型转换 |
| 平台特定行为 | 检查是否依赖特定OS/运行时版本 |
| 时序依赖 | 检查是否依赖执行速度/时间 |

---

## 反模式

- 🚫 不读现有代码就动手改（理解是修改的前提）
- 🚫 顺手重构"看起来不好"的代码（最小改动原则）
- 🚫 删除看不懂的代码（可能是隐式契约）
- 🚫 不加测试就改遗留代码（改完无法验证）
