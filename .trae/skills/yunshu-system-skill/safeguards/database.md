## N+1查询防护（P-021核心）

### N+1查询识别

| 模式 | 具体表现 | 正确写法 |
|------|----------|----------|
| 循环中SELECT | `for (id of ids) { db.find(id) }` | `db.findAll({ where: { id: ids } })` |
| 关联查询N+1 | 查主表后逐条查关联表 | JOIN/eager loading/prefetch |
| 批量操作逐条执行 | 循环中逐条INSERT/UPDATE | 批量INSERT/UPDATE |
| 未使用索引 | WHERE条件字段无索引 | 添加索引或使用已索引字段 |

### 检查规则

每次生成数据库查询代码时：

1. **循环检查**：是否有数据库查询在循环内？→ 必须改为批量查询
2. **关联检查**：是否查主表后逐条查关联表？→ 必须使用JOIN或eager loading
3. **索引检查**：WHERE/JOIN条件字段是否有索引？→ 建议添加索引
4. **分页检查**：是否查询全表数据？→ 必须添加分页限制
5. **事务检查**：多步操作是否需要事务？→ 涉及多表写操作必须用事务

### ORM特定检查

| ORM | 常见N+1陷阱 | 正确写法 |
|-----|------------|----------|
| Sequelize | `include`未加`separate:true` | 使用`separate:true`或手动JOIN |
| TypeORM | `find`未加`relations` | 使用`relations`选项或`QueryBuilder` |
| Prisma | `include`嵌套过深 | 使用`select`精确选择字段 |
| SQLAlchemy | `lazy='dynamic'`循环调用 | 使用`joinedload()`或`selectinload()` |

## API版本控制（P-025核心）

### 版本控制策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| URL版本 | `/api/v1/users` | 公开API，客户端多样 |
| Header版本 | `Accept: application/vnd.api.v1+json` | 内部API，客户端可控 |
| 参数版本 | `/api/users?version=1` | 简单场景，不推荐 |

### API变更兼容性规则

| 变更类型 | 兼容性 | 处理方式 |
|----------|--------|----------|
| 新增端点 | ✅ 兼容 | 直接添加 |
| 新增字段 | ✅ 兼容 | 直接添加，旧客户端忽略 |
| 新增可选参数 | ✅ 兼容 | 添加默认值 |
| 删除字段 | ❌ 破坏 | 新版本中删除，旧版本保留 |
| 修改字段类型 | ❌ 破坏 | 新版本中修改，旧版本保留 |
| 重命名端点 | ❌ 破坏 | 旧端点重定向到新端点 |

### API设计检查清单

1. 每个API端点是否有版本标识？
2. 破坏性变更是否在新版本中？
3. 旧版本是否有废弃计划和下线时间？
4. API文档是否与代码同步？

## 反模式

- 🚫 循环中执行数据库查询（N+1问题）
- 🚫 查全表不分页（数据量大了直接OOM）
- 🚫 多表写操作不用事务（数据不一致）
- 🚫 WHERE条件字段无索引（全表扫描）
- 🚫 API变更不考虑向后兼容（客户端直接崩溃）
