> 覆盖问题：P-062 文档漂移 / P-063 架构决策无记录 / P-064 隐性知识缺失 / P-065 只写代码不写文档 / P-066 重复代码 / P-068 短期变动激增
> 触发场景：代码执行时（03-execute）、代码交付时（05-deliver）
> 加载方式：由03-execute和05-deliver组件检测到代码维护场景时触发读取本文件

## 第一层：文档同步机制（P-062/P-065）

> 在03-execute阶段触发，确保代码变更时同步更新文档。

### 文档同步检查清单

| 变更类型 | 必须同步的文档 | 检查方式 |
|----------|---------------|----------|
| 新增函数/方法 | 函数注释+API文档 | 函数签名与文档描述是否一致 |
| 修改函数签名 | 调用方文档+API文档 | 参数/返回值是否同步更新 |
| 新增配置项 | 配置文档+README | 配置说明是否同步更新 |
| 修改数据模型 | 数据文档+迁移说明 | 字段变更是否记录 |
| 新增依赖 | 依赖文档+安装说明 | 依赖列表是否同步更新 |
| 修改环境变量 | 环境配置文档 | .env.example是否同步 |
| 修改API端点 | API文档+集成文档 | 接口变更是否同步 |

### 文档同步门禁流程

```
代码变更完成
  ↓
1. 识别变更类型（新增/修改/删除）
  ↓
2. 对照文档同步检查清单，确定需要同步的文档
  ↓
3. 检查文档是否已同步更新
  ├─ 已同步 → 继续
  ├─ 未同步 → 提醒同步，暂停进入下一步
  └─ 无对应文档 → 创建文档
  ↓
4. 记录文档同步状态到变更日志
```

### AI文档生成规则（P-065）

| 代码产物 | 必须生成的文档 | 格式要求 |
|----------|---------------|----------|
| 新模块 | 模块说明文档 | 目的+职责+接口+依赖 |
| 新函数 | 函数注释 | 参数+返回值+异常+示例 |
| 新API | API文档 | 端点+方法+参数+响应+错误码 |
| 新配置 | 配置说明 | 名称+类型+默认值+说明 |
| 新类型 | 类型文档 | 字段+约束+示例 |

## 第二层：决策记录与知识持久化（P-063/P-064）

> 在03-execute和05-deliver阶段触发，确保架构决策有记录、隐性知识能持久化。

### ADR强制记录规则（P-063）

以下场景**必须**记录ADR：

| 场景 | ADR内容 | 优先级 |
|------|---------|--------|
| 选择技术栈 | 为什么选A不选B | P0 |
| 模块拆分决策 | 为什么这样拆 | P0 |
| 数据模型设计 | 为什么这样建模 | P0 |
| API设计决策 | 为什么这样设计接口 | P1 |
| 性能优化方案 | 为什么选这个方案 | P1 |
| 安全方案选择 | 为什么这样实现安全 | P0 |
| 第三方服务选择 | 为什么选这个服务 | P1 |

### ADR模板（复用understanding.md中的模板）

```markdown
### ADR-XXX: [决策标题]

**背景**：为什么需要做这个决策？
**选项**：
- A: [选项A描述] — 优点/缺点
- B: [选项B描述] — 优点/缺点
**决策**：选择[X]，因为[原因]
**影响**：这个决策会影响哪些模块/功能？
**可逆性**：如果决策错误，回退成本多大？
```

### 隐性知识捕获规则（P-064）

| 隐性知识类型 | 捕获方式 | 存储位置 |
|-------------|----------|----------|
| 业务规则 | 在代码注释中标注`@business-rule` | 代码+文档 |
| 设计意图 | 在ADR中记录"为什么" | ADR文档 |
| 历史决策 | 在变更日志中记录决策原因 | 变更日志 |
| 已知限制 | 在代码注释中标注`@known-limitation` | 代码+文档 |
| 性能约束 | 在代码注释中标注`@performance-constraint` | 代码+文档 |
| 安全假设 | 在代码注释中标注`@security-assumption` | 代码+文档 |

## 第三层：代码去重与变动管控（P-066/P-068）

> 在03-execute阶段触发，防止重复代码和变动激增。

### 重复代码检测规则（P-066）

| 检测时机 | 检测方式 | 处理方式 |
|----------|----------|----------|
| 新增函数时 | 搜索相似函数名/相似逻辑 | 相似度>70% → 提醒抽取公共函数 |
| 新增文件时 | 搜索相似文件结构 | 相似度>60% → 提醒合并模块 |
| 修改代码时 | 检查是否有其他地方需要同步修改 | 有 → 标记为重复代码，建议抽取 |
| 代码审查时 | 检查同类功能的实现方式 | 实现方式不一致 → 统一实现 |

### 重复代码处理策略

| 重复类型 | 处理方式 | 优先级 |
|----------|----------|--------|
| 完全重复（复制粘贴） | 抽取公共函数/工具类 | P0 |
| 逻辑重复（实现不同但逻辑相同） | 统一实现方式 | P1 |
| 结构重复（代码结构相似） | 抽象模板/基类 | P2 |
| 配置重复（配置项重复） | 抽取配置常量 | P1 |

### 短期变动管控（P-068）

| 管控维度 | 限制规则 | 说明 |
|----------|----------|------|
| 单次变更文件数 | ≤10个 | 超过需拆分为多次变更 |
| 单次变更行数 | ≤500行 | 超过需分批提交 |
| 单日变更次数 | ≤5次重大变更 | 超过需评估维护负担 |
| 变更影响范围 | 必须记录影响矩阵 | 防止隐性影响扩散 |
| 变更回滚能力 | 每次变更必须有回滚方案 | 防止不可逆变更 |

### 变动节奏建议

```
AI辅助开发节奏：
  ├─ 编码阶段：可以快速迭代
  ├─ 审查阶段：必须放慢速度，逐行审查
  ├─ 测试阶段：必须充分测试，不能跳步
  └─ 交付阶段：必须完整记录，不能遗漏文档

节奏切换信号：
  ├─ 连续3次快速变更 → 提醒放慢审查
  ├─ 变更文件>10个 → 建议拆分提交
  └─ 变更行数>500行 → 建议分批交付
```

- 🚫 代码改了文档没改（P-062文档漂移）

- 🚫 连续快速变更不审查（P-068节奏失控）
