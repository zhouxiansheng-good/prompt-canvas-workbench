## 第二层：架构决策检查（P-016）

> 在02-plan阶段触发，确保AI不会盲目套模板，而是基于业务特征做架构决策。

### 架构决策检查清单

| 决策维度 | 检查问题 | 常见AI错误 |
|----------|----------|-----------|
| **模块拆分** | 功能边界在哪？哪些应该独立模块？ | 把所有功能塞进一个文件/模块 |
| **技术选型** | 为什么选这个技术栈？有没有更合适的？ | 无脑用最流行的框架，不管是否适合 |
| **数据边界** | 数据归属哪个模块？跨模块数据怎么流转？ | 到处直接操作数据，没有清晰的归属 |
| **接口设计** | 模块间怎么通信？同步还是异步？ | 模块间紧耦合，没有明确接口 |
| **扩展性** | 未来可能怎么变？现在要预留什么？ | 只满足当前需求，无法扩展 |
| **部署单元** | 哪些需要独立部署？哪些可以合并？ | 全部打包或全部拆微服务，走极端 |

### 架构决策记录模板

每个重要架构决策必须记录：

```markdown
### ADR-XXX: [决策标题]

**背景**：为什么需要做这个决策？
**选项**：
- A: [选项A描述] — 优点/缺点
- B: [选项B描述] — 优点/缺点
**决策**：选择[X]，因为[原因]
**影响**：这个决策会影响哪些模块/功能？
**可逆性**：如果决策错误，回退成本多大？
```

### 架构反模式检测

| 反模式 | 检测信号 | 修正方式 |
|--------|----------|----------|
| 大泥球 | 单文件>500行，函数间无清晰边界 | 按职责拆分模块 |
| 过度设计 | 3个接口+2个抽象类+1个工厂，只为一个简单功能 | YAGNI，只做当前需要的 |
| 技术栈混乱 | 同一项目混用3种状态管理方案 | 统一技术栈，参考dependency.md |
| 数据无主 | 多个模块直接读写同一数据 | 确定数据归属，通过接口访问 |
| 同步地狱 | 所有模块间调用都是同步的 | 识别可异步的场景，引入消息队列 |

## 和AI协作时的好习惯（架构相关）

| 习惯 | 说明 | 反模式 |
|------|------|--------|
| 要求解释 | 让AI解释为什么这样设计 | 盲目接受AI的架构方案 |
| 保留决策记录 | 重要决策写ADR文档 | 决策只在对话中，没有持久化 |
| 分步确认 | 每个阶段确认AI的理解是否正确 | 一次把所有需求说完就不管了 |
