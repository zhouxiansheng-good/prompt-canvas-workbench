# 主观偏差校准

> 覆盖问题：P-054 主观偏差 / P-061 监督者角色局限
> 触发场景：任务启动时（01-init）
> 加载方式：01-init专项评估门禁触发

---

## 问题描述

| 编号 | 问题 | 严重程度 | 具体表现 |
|------|------|---------|---------|
| P-054 | 主观偏差39个百分点 | 🔴 致命 | 被AI减速19%后，仍坚信自己快了20% |
| P-061 | 监督者角色局限 | 🟠 严重 | 人类从操作者变成监督者，难以保持专注 |

---

## 偏差校准检查

| 偏差类型 | 具体表现 | 校准方式 |
|----------|----------|----------|
| 速度高估 | 觉得AI让自己快了，实际可能更慢 | 记录实际耗时vs预估耗时，定期对比 |
| 质量高估 | 觉得AI代码质量高，实际缺陷密度2-3.5倍 | 运行代码质量检查，用数据说话 |
| 理解高估 | 觉得自己理解了AI代码，实际没懂 | 执行理解验证门禁 |
| 信任过度 | 觉得AI不会出错 | 展示AI常见错误模式（见code-quality.md） |

### 监督者角色防护（P-061）

| 风险 | 具体表现 | 防护方式 |
|------|----------|----------|
| 注意力涣散 | 监督者走神，AI输出什么就通过什么 | 关键节点强制理解验证 |
| 责任感稀释 | "AI写的，不是我负责" | 代码审查签名：开发者确认理解并负责 |
| 审查疲劳 | AI生成速度远超审查速度 | 分批审查，每批不超过5个文件 |
| 能力脱节 | 只审查不理解，逐渐看不懂代码 | 定期手写代码练习 |

### 效率度量真实化

```
每次任务完成后记录：
  ├─ 预估耗时：[开发者预估]
  ├─ 实际耗时：[真实计时]
  ├─ AI辅助比例：[AI生成代码占比]
  ├─ 返工次数：[因AI代码问题返工的次数]
  └─ 理解验证：[通过/未通过]
```

## 防偏差的好习惯

| 习惯 | 说明 | 反模式 |
|------|------|--------|
| 计时对比 | 记录预估vs实际耗时 | 凭感觉判断效率 |
| 质量度量 | 用代码检查工具量化质量 | "看起来没问题" |
| 理解自检 | 每完成一个功能自问"我能解释吗" | 跳过理解直接交付 |
| 承认局限 | 承认AI有局限，需要人工审查 | 盲目信任AI |

---

## 反模式

- 🚫 觉得AI让自己快了20%但实际慢了19%（P-054主观偏差）
- 🚫 AI输出什么就通过什么（P-061监督者走神）
- 🚫 觉得"AI写的，不是我负责"（P-061责任感稀释）
- 🚫 不记录实际耗时凭感觉判断效率（P-054偏差）
