# 代码可持续性防护

> 覆盖问题：P-067 "有效直到无效"（代码黑箱） / P-069 长期存活率低
> 触发场景：代码交付时（05-deliver）、验收时（04-accept）
> 加载方式：由05-deliver和04-accept组件检测到交付/验收场景时触发读取本文件

---

## 问题描述

| 编号 | 问题 | 严重程度 | 具体表现 |
|------|------|---------|---------|
| P-067 | "有效直到无效" | 🔴 致命 | 代码不工作时你不懂原因也不知能否修复，完全变成黑箱 |
| P-069 | 长期存活率低 | 🔴 致命 | AI长期存活率测试成功率低于15%，项目不可持续 |

## 第一层：黑箱消除机制（P-067）

> 在05-deliver阶段触发，确保交付的代码不是黑箱，任何开发者都能理解和维护。

### 黑箱诊断清单

| 黑箱信号 | 诊断问题 | 严重度 |
|----------|----------|--------|
| 无决策记录 | "为什么这样写而不是那样写？" — 没人知道 | 🔴 |
| 无错误地图 | "这段代码会怎么失败？失败后怎么办？" — 没人知道 | 🔴 |
| 无数据流向 | "数据从哪来？到哪去？中间怎么变？" — 没人知道 | 🔴 |
| 无修改指南 | "如果要加功能/改逻辑，应该改哪里？" — 没人知道 | 🔴 |
| 无依赖说明 | "这段代码依赖什么？被谁依赖？" — 没人知道 | 🟠 |
| 无测试覆盖 | "怎么验证这段代码是对的？" — 没人知道 | 🔴 |

### 黑箱消除产物

每个交付的代码模块必须包含以下产物：

| 产物 | 内容 | 模板 |
|------|------|------|
| **决策记录** | 每个重要设计决策的"为什么" | ADR模板（见understanding.md） |
| **错误地图** | 每个可能失败点的处理方式 | 错误地图模板 |
| **数据流向图** | 数据从输入到输出的完整路径 | 数据流向模板 |
| **修改指南** | 常见修改场景的操作指引 | 修改指南模板 |
| **依赖关系** | 模块依赖和被依赖关系 | 依赖矩阵 |

### 错误地图模板

```markdown
### 错误地图：[模块名]

| 失败点 | 触发条件 | 表现 | 处理方式 | 恢复策略 |
|--------|----------|------|----------|----------|
| [失败点1] | [什么情况下会失败] | [失败时什么表现] | [代码怎么处理] | [如何恢复] |
```

### 数据流向模板

```markdown
### 数据流向：[模块名]

输入：[数据来源] → [格式/类型/约束]
  ↓
处理步骤1：[做什么] → [数据变成什么]
  ↓
处理步骤2：[做什么] → [数据变成什么]
  ↓
输出：[数据去向] → [格式/类型/约束]

异常分支：[异常输入] → [处理方式] → [输出]
```

### 修改指南模板

```markdown
### 修改指南：[模块名]

| 修改场景 | 要改的文件 | 要改的函数 | 注意事项 | 影响范围 |
|----------|-----------|-----------|----------|----------|
| 添加新字段 | [文件:行] | [函数名] | [注意] | [影响] |
| 修改验证规则 | [文件:行] | [函数名] | [注意] | [影响] |
| 更换数据源 | [文件:行] | [函数名] | [注意] | [影响] |
```

### 黑箱评分

| 评分 | 标准 | 处理方式 |
|------|------|----------|
| 🟢 透明 | 5个产物齐全，任何开发者都能理解和修改 | 可以交付 |
| 🟡 半透明 | 3-4个产物齐全，核心逻辑可理解 | 补充缺失产物后交付 |
| 🔴 黑箱 | <3个产物，核心逻辑不可理解 | 不许交付，必须补充 |

## 第二层：可持续性评估（P-069）

> 在04-accept阶段触发，评估代码的长期可持续性，预测存活概率。

### 可持续性评估维度

| 维度 | 评估问题 | 权重 | 评分标准 |
|------|----------|------|----------|
| **可理解性** | 新人能否在30分钟内理解核心逻辑？ | 25% | 🟢能 / 🟡需1小时 / 🔴看不懂 |
| **可修改性** | 修改一个功能需要改几个文件？ | 20% | 🟢1-2个 / 🟡3-5个 / 🔴>5个 |
| **可测试性** | 关键路径是否有自动化测试？ | 20% | 🟢全覆盖 / 🟡部分覆盖 / 🔴无测试 |
| **可恢复性** | 失败后能否自动恢复或快速定位？ | 15% | 🟢自动恢复 / 🟡可定位 / 🔴无法定位 |
| **可维护性** | 代码是否有明确的维护入口？ | 10% | 🟢有文档+注释 / 🟡有注释 / 🔴裸代码 |
| **可演进性** | 架构是否支持未来扩展？ | 10% | 🟢预留扩展点 / 🟡可扩展 / 🔴硬编码 |

### 可持续性评分计算

```
总分 = Σ(维度评分 × 权重)

🟢 = 3分, 🟡 = 2分, 🔴 = 1分

可持续性等级：
  ≥2.5 → 🟢 可持续（预计存活率>60%）
  2.0~2.5 → 🟡 需改进（预计存活率30-60%）
  <2.0 → 🔴 不可持续（预计存活率<30%）
```

### 不可持续代码的修复优先级

| 评分维度 | 🔴时的修复动作 | 优先级 |
|----------|---------------|--------|
| 可理解性 | 补充决策记录+数据流向图 | P0 |
| 可测试性 | 补充关键路径测试 | P0 |
| 可修改性 | 拆分大函数/大文件 | P1 |
| 可恢复性 | 补充错误处理和恢复策略 | P1 |
| 可维护性 | 补充文档和关键注释 | P2 |
| 可演进性 | 识别扩展点，预留接口 | P2 |

### 长期存活率提升策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 决策持久化 | 所有设计决策写入ADR文档 | 所有项目 |
| 测试先行 | 关键路径必须有测试 | 所有项目 |
| 文档同步 | 代码变更时同步更新文档 | 所有项目 |
| 定期回顾 | 每月回顾代码健康度 | 长期项目 |
| 渐进重构 | 发现技术债务时及时重构 | 维护期项目 |

## 反模式

- 🚫 交付代码没有决策记录（P-067黑箱）
- 🚫 交付代码没有错误地图（P-067黑箱）
- 🚫 交付代码没有数据流向图（P-067黑箱）
- 🚫 关键路径没有自动化测试（P-069不可持续）
- 🚫 代码修改需要改>5个文件（P-069紧耦合）
- 🚫 失败后无法自动恢复或快速定位（P-069不可恢复）
- 🚫 新人看不懂核心逻辑（P-067黑箱+P-069不可理解）
- 🚫 代码变更时不同步更新文档（P-069文档漂移）
