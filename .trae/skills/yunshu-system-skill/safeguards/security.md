## AI代码常见安全漏洞模式

### 第一类：注入攻击（P-011核心）

| 漏洞模式 | AI典型错误 | 安全写法 | 严重度 |
|----------|-----------|----------|--------|
| SQL注入 | 字符串拼接SQL | 参数化查询/ORM | 🔴 |
| 命令注入 | 直接拼接用户输入到shell命令 | 使用安全API，禁止拼接 | 🔴 |
| XSS | 直接渲染用户输入为HTML | 转义/使用安全渲染框架 | 🔴 |
| 路径遍历 | 直接拼接用户输入到文件路径 | 白名单校验/路径规范化 | 🔴 |
| LDAP注入 | 直接拼接用户输入到LDAP查询 | 参数化LDAP查询 | 🟠 |

### 第二类：认证授权缺陷（P-013核心）

| 漏洞模式 | AI典型错误 | 安全写法 | 严重度 |
|----------|-----------|----------|--------|
| 硬编码密钥 | API Key/Secret直接写在代码中 | 环境变量/密钥管理服务 | 🔴 |
| 弱密码策略 | 默认密码/无密码验证 | 强密码策略+盐值哈希 | 🔴 |
| 权限缺失 | 只检查登录不检查权限 | 每个操作都检查RBAC权限 | 🔴 |
| 越权访问 | 只验证用户身份不验证资源归属 | 验证当前用户是否有权操作该资源 | 🔴 |
| Session固定 | 登录后不更新Session ID | 登录成功后重新生成Session | 🟠 |
| JWT不安全 | 不验证签名/使用none算法/密钥过弱 | 严格验证签名+算法+过期时间 | 🔴 |

### 第三类：敏感信息泄露（P-011核心）

| 漏洞模式 | AI典型错误 | 安全写法 | 严重度 |
|----------|-----------|----------|--------|
| 日志泄露 | 将密码/Token打印到日志 | 脱敏处理/不记录敏感字段 | 🔴 |
| 错误信息泄露 | 将堆栈信息/SQL直接返回前端 | 通用错误提示+服务端详细日志 | 🟠 |
| 注释泄露 | 代码注释中包含密钥/内部URL | 清理所有敏感注释 | 🔴 |
| 响应泄露 | API返回过多字段（含内部字段） | 只返回必要字段/DTO过滤 | 🟠 |

### 第四类：DevOps配置安全（P-026核心）

| 漏洞模式 | AI典型错误 | 安全写法 | 严重度 |
|----------|-----------|----------|--------|
| Docker以root运行 | Dockerfile中未指定USER | 添加非root用户 | 🔴 |
| 环境变量明文 | 密钥直接写在docker-compose.yml | 使用密钥管理服务 | 🔴 |
| 端口全开 | EXPOSE所有端口 | 只暴露必要端口 | 🟠 |
| CI权限过大 | pipeline有完整仓库写权限 | 最小权限原则 | 🟠 |
| 依赖固定缺失 | 不锁定依赖版本 | 使用lock文件固定版本 | 🟠 |

## 安全编码规则

> 涉及安全相关代码时，必须遵守以下规则。

### 铁律（违反即停止）

1. **绝不硬编码密钥**：所有密钥/Token/密码必须通过环境变量或密钥管理服务获取
2. **绝不拼接用户输入**：SQL/命令/路径/URL中禁止直接拼接用户输入
3. **绝不信任前端数据**：所有权限校验必须在服务端执行
4. **绝不静默吞错**：安全相关错误必须记录日志并适当响应

### 安全检查流程

当检测到安全相关代码时：

```
1. 识别安全类别（注入/认证/泄露/DevOps）
2. 对照对应漏洞模式清单逐项检查
3. 发现漏洞 → 标记严重度 → 修复后重新检查
4. 记录安全检查结果
```

### 安全检查结果记录

```yaml
security_check:
  category: injection/auth/leakage/devops
  triggers_found: [触发的关键词列表]
  vulnerabilities: [发现的漏洞列表]
  critical_count: N
  action: pass/fix_required/block
  notes: "<备注>"
```

### 判定标准

| 严重漏洞数 | 判定 | 动作 |
|-----------|------|------|
| 0 | ✅ 通过 | 继续下一步 |
| 0（仅有🟠） | ⚠️ 轻微 | 建议修复，可继续 |
| ≥1 | ❌ 不通过 | 必须修复后重新检查 |

## 权限提升防护（P-013专项）

### 权限检查矩阵

每个API端点/操作必须明确：

| 操作 | 需要登录？ | 需要的角色 | 资源归属检查 |
|------|-----------|-----------|-------------|
| | 是/否 | 角色列表 | 是/否 |

### 常见权限提升路径

| 攻击路径 | AI常见疏忽 | 防护措施 |
|----------|-----------|----------|
| 水平越权 | 只检查登录不检查资源归属 | 每个操作验证资源userId==当前userId |
| 垂直越权 | 普通用户可访问管理员接口 | 每个接口检查角色权限 |
| IDOR | 可通过修改ID访问他人资源 | 使用间接引用+权限校验 |
| 权限绕过 | API有权限但静态资源无权限 | 静态资源也需权限控制 |

## 反模式

- 🚫 硬编码密钥/Token/密码（哪怕注释里也不行）
- 🚫 拼接用户输入到SQL/命令/路径（参数化查询是唯一安全方式）
- 🚫 只检查登录不检查权限（登录≠授权）
- 🚫 将敏感信息写入日志/注释/错误响应
- 🚫 Docker以root运行/端口全开/密钥明文
- 🚫 跳过安全检查直接提交（安全漏洞不会自己消失）
