## 跨文件重构安全流程（P-018核心）

### 重构前必做：影响分析

```
1. 识别变更点：具体改哪个文件/函数/变量？
2. 识别调用方：谁在调用这个函数/使用这个变量？（Grep搜索）
3. 识别依赖方：谁依赖这个模块/文件？（import搜索）
4. 构建影响矩阵：
   | 变更点 | 调用方 | 依赖方 | 影响程度 | 需要同步修改？ |
   |--------|--------|--------|----------|---------------|
   |        |        |        | 高/中/低 | 是/否         |
5. 评估重构范围：单文件/跨文件/跨模块/跨项目
```

### 重构执行策略

| 重构范围 | 策略 | 验证要求 |
|----------|------|----------|
| 单文件 | 直接修改 | 修改后跑该文件测试 |
| 跨文件（2-3个） | 逐文件修改，每改一个跑测试 | 每步验证 |
| 跨模块（3-5个） | 分阶段重构，每阶段独立验证 | 每阶段跑全量测试 |
| 跨项目（>5个） | 必须拆分为多次重构 | 每次有独立回滚点 |

### 重构后必做：回归验证

1. 跑全量测试（不只是修改模块的测试）
2. 检查所有调用方是否正常工作
3. 检查import关系是否正确
4. 确认没有遗留的旧引用

## 循环依赖防护（P-019核心）

### 循环依赖检测

每次新增import时，检查是否形成循环：

```
A imports B
B imports C
C imports A  ← 循环依赖！
```

**检测方法**：
1. 重构前画出当前模块依赖图
2. 新增import后检查是否形成环
3. 发现环 → 必须重构消除循环

### 循环依赖消除策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 接口抽取 | 将共享逻辑抽取到独立模块，A和B都依赖新模块 | 双向依赖 |
| 事件解耦 | A发事件，B监听事件，不直接import | 松耦合场景 |
| 依赖注入 | A通过接口依赖B，B不依赖A | 控制反转 |
| 层级分离 | 严格分层，上层依赖下层，下层不依赖上层 | 架构层面 |

### 依赖方向规则

- ✅ 允许：高层→低层、业务→基础、页面→组件→工具
- ❌ 禁止：反向依赖、跨层依赖、同层互依赖

## 接口契约保护（P-020核心）

### 接口变更检查清单

每次修改函数签名/类型定义/接口时：

| 检查项 | 说明 | 必须通过 |
|--------|------|----------|
| 向后兼容 | 新参数是否有默认值？ | 是 |
| 返回值兼容 | 返回类型是否扩展而非缩减？ | 是 |
| 删除检查 | 删除的属性/方法是否还有调用方？ | 无调用方才可删 |
| 重命名检查 | 重命名的属性/方法是否所有引用都已更新？ | 全部更新 |
| 类型缩窄检查 | 类型是否从宽变窄（如any→string）？ | 需评估影响 |

### 接口变更策略

| 变更类型 | 策略 |
|----------|------|
| 新增字段/参数 | ✅ 安全，加默认值 |
| 扩展返回类型 | ✅ 安全，新增字段 |
| 重命名字段 | ⚠️ 需两步：先新增→迁移调用方→再删除旧 |
| 删除字段 | ❌ 高危，必须确认无调用方 |
| 修改类型 | ⚠️ 需评估所有调用方 |

### 接口语义一致性检查

修改接口后，不仅检查类型是否匹配，还要检查语义是否一致：

| 表面匹配 | 实际不匹配 |
|----------|-----------|
| 返回`User[]` | 旧版返回所有用户，新版只返回活跃用户 |
| 参数`limit: number` | 旧版是最大数量，新版是分页大小 |
| 返回`{ success: boolean }` | 旧版表示操作成功，新版表示数据存在 |

## 反模式

- 🚫 跨文件重构不画影响矩阵（30%成功率不是开玩笑的）
- 🚫 新增import不检查循环依赖
- 🚫 修改接口不检查所有调用方
- 🚫 重构后只跑修改模块的测试（必须跑全量测试）
- 🚫 一次重构跨太多文件（分阶段，每阶段可回滚）
- 🚫 接口重命名一步到位（应两步：新增→迁移→删除旧）
