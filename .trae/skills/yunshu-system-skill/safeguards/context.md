# 上下文效率防护

> 覆盖问题：P-001 上下文窗口限制 / P-002 注意力稀释 / P-004 信息过载 / P-005 上下文污染
> 触发场景：会话过长/读取大量文件/上下文接近容量上限时
> 加载方式：由01-init Step 0恢复场景和03-execute门控检查触发读取本文件
> 与P-003关系：P-003解决会话级退化（何时换会话），本文件解决交互级效率（如何用好上下文）

---

## 问题描述

| 编号 | 问题 | 严重程度 | 具体表现 |
|------|------|---------|---------|
| P-001 | 上下文窗口限制 | 🔴 致命 | 上下文有上限，信息塞不下 |
| P-002 | 注意力稀释 | 🔴 致命 | 信息都在但AI只关注最近的，忽略早期关键约束 |
| P-004 | 信息过载 | 🟠 严重 | 一次性读取太多文件，关键信息被淹没 |
| P-005 | 上下文污染 | 🟠 严重 | 错误信息/过时信息残留在上下文中，误导后续决策 |

---

## 上下文利用效率规则

### 规则1：分层加载，按需读取

> 不要一次性读取所有文件，按优先级分层加载。

| 层级 | 内容 | 何时加载 | 加载方式 |
|------|------|----------|----------|
| L0 | 任务卡+当前阶段目标 | 会话开始时 | 全量加载 |
| L1 | 与当前任务直接相关的文件 | 执行每个任务前 | 按需读取 |
| L2 | 与当前任务间接相关的文件 | 需要理解上下文时 | 按需读取 |
| L3 | 历史决策和约束 | 需要回溯时 | 从阶段记忆卡/交接文档读取 |

**禁止行为**：
- 🚫 一次性读取整个项目目录
- 🚫 读取与当前任务无关的文件"以防万一"
- 🚫 将大文件全文加载到上下文中（应只读相关部分）

### 规则1.5：上下文账本优先

> 每次调查后留下可复用的账本，下一轮先验证账本新鲜度，再决定补读范围。

**启动流程**：

1. `python scripts/yunshu.py context list --task-id <task_id>`
2. `python scripts/yunshu.py context show <context_id>`
3. `python scripts/yunshu.py context status <context_id>`
4. 对 fresh 来源复用账本结论；对 stale/missing 来源补读原文；对账本未覆盖的新问题按规则1分层读取

**记录流程**：

```bash
python scripts/yunshu.py context record \
  --task-id <task_id> \
  --phase <phase> \
  --source <path-or-url> \
  --finding "<关键发现>" \
  --action "<已完成动作>" \
  --gap "<缺口>" \
  --next-read "<下次补读>"
```

**反模式**：
- 🚫 有 fresh 账本仍然重读全部历史
- 🚫 只写“我读过了”而不记录具体来源
- 🚫 只记录来源不记录结论、动作、缺口
- 🚫 文件已 stale 仍直接沿用旧结论

### 规则2：注意力聚焦策略

> AI的注意力在长上下文中会稀释，必须主动聚焦关键信息。

**聚焦方法**：

| 方法 | 说明 | 使用时机 |
|------|------|----------|
| 关键约束前置 | 在每次回复前，先回顾任务卡中的关键约束 | 每个任务开始时 |
| 决策锚定 | 将已确认的决策标记为「锚定决策」，后续不可遗忘 | 做出重要决策时 |
| 上下文摘要 | 每完成一个阶段，对上下文做摘要压缩 | 阶段过渡时 |
| 信号放大 | 将关键信息用结构化格式（表格/清单）呈现，而非散落在段落中 | 记录关键信息时 |

**注意力稀释信号**：

| 信号 | 具体表现 | 修复动作 |
|------|----------|----------|
| 忽略早期约束 | 违反了任务卡中已声明的约束 | 重新读取任务卡，标记约束 |
| 遗忘已确认决策 | 重复讨论已确认的问题 | 读取决策锚定记录 |
| 回复冗长 | 输出大量无关信息 | 聚焦当前任务，删除无关内容 |
| 上下文跳跃 | 在多个话题间跳跃而不深入 | 回到当前任务，一次只做一件事 |

### 规则3：信息过载防护

> 读取文件前先评估必要性，避免信息过载。

**文件读取决策树**：

```
需要读取文件？
  ↓
1. 这个文件与当前任务直接相关吗？
   - 是 → 继续
   - 否 → 不读取
  ↓
2. 需要读取全文还是部分？
   - 文件<100行 → 全文
   - 文件100-500行 → 读取相关部分（指定offset和limit）
   - 文件>500行 → 先用Grep搜索关键词，只读匹配部分
  ↓
3. 读取后是否需要保留在上下文中？
   - 一次性参考 → 读取后可压缩
   - 持续参考 → 保留关键部分
```

**批量读取上限**：
- 单次最多同时读取3个文件
- 超过3个 → 分批读取，先读最关键的
- 每个文件读取后立即提取关键信息，避免全文残留

### 规则4：上下文污染清理

> 过时信息和错误信息必须主动清理。

**污染源与清理方法**：

| 污染源 | 清理方法 |
|--------|----------|
| 已废弃的方案讨论 | 在阶段记忆卡中标记为「已废弃」，后续不再引用 |
| 已修复的Bug描述 | Bug修复后从活跃上下文中移除，存入bug_knowledge.md |
| 已过时的代码片段 | 不在上下文中保留旧代码，只保留当前版本 |
| 探索性讨论的中间产物 | 探索完成后只保留结论，删除中间过程 |
| 错误的假设 | 明确标记为「已证伪」，防止后续引用 |

**清理时机**：
- 每个任务完成后：清理该任务的临时信息
- 阶段过渡时：压缩上一阶段上下文
- 发现矛盾时：检查是否有过时信息导致矛盾

## 上下文容量评估

> 无法精确计数token，但可以通过以下指标评估上下文使用情况。

### 容量评估指标

| 指标 | 评估方法 | 阈值 |
|------|----------|------|
| 已读取文件数 | 统计本次会话读取的文件数 | >10个 → 注意 |
| 已完成交互轮次 | 统计本次会话的用户消息数 | >15轮 → 注意 |
| 已完成任务数 | 统计TodoWrite中已完成的任务 | >5个 → 注意 |
| 重复读取同一文件 | 同一文件被读取>2次 | → 可能上下文已稀释 |

### 容量管理动作

| 评估结果 | 动作 |
|----------|------|
| 健康（所有指标低于阈值） | 正常继续 |
| 注意（1个指标超阈值） | 执行上下文摘要压缩 |
| 警告（≥2个指标超阈值） | 执行压缩 + 评估是否需要换会话（触发P-003会话健康规则） |

## 与P-003会话健康规则的协作

| 场景 | 本文件负责 | P-003负责 |
|------|-----------|-----------|
| 单次交互信息过载 | 分层加载+按需读取 | — |
| 注意力在单次交互中稀释 | 聚焦策略+约束前置 | — |
| 上下文污染 | 主动清理过时信息 | — |
| 会话级退化 | 评估容量指标 | 门控检查+换会话 |
| 换会话后恢复 | — | 交接文档+上下文包 |

## 反模式

- 🚫 一次性读取整个项目目录（信息过载）
- 🚫 读取与当前任务无关的文件（浪费上下文空间）
- 🚫 不做摘要就让上下文自然膨胀（注意力稀释）
- 🚫 保留已废弃的方案讨论在上下文中（上下文污染）
- 🚫 忽略早期约束只关注最近对话（注意力稀释最常见表现）
- 🚫 大文件全文加载（应分段读取或Grep搜索）
