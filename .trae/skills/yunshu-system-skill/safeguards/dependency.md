## 3步验证流程

引入任何新依赖前，必须按顺序执行以下3步验证：

### Step 1：包名存在性验证

**目标**：确认包名在对应包管理器中真实存在。

**方法**：

| 语言 | 验证命令 | 通过条件 |
|------|----------|----------|
| Node.js | `npm view <包名>` | 返回包信息（非404） |
| Python | `pip index versions <包名>` | 返回版本列表 |
| Rust | `cargo search <包名>` | 搜索结果中包含该包 |
| Go | `go list -m <包名>` | 返回模块路径 |
| Java | 检查Maven Central搜索 | 包存在 |

**验证失败处理**：
1. 包名不存在 → 标记为「幻觉依赖」
2. 搜索相似包名 → 提供2-3个候选
3. 用户确认正确包名后才可继续

**自检规则**：
- 检查包名拼写是否与官方文档一致
- 检查是否混淆了相似包名（如 `lodash` vs `underscore`）
- 检查是否使用了项目已有依赖的别名

### Step 2：版本可用性验证

**目标**：确认指定的版本号存在且可用，避免使用过时或废弃版本。

**方法**：

| 语言 | 验证命令 | 通过条件 |
|------|----------|----------|
| Node.js | `npm view <包名> versions` | 指定版本在列表中 |
| Python | `pip index versions <包名>` | 指定版本在列表中 |
| Rust | `cargo search <包名>` | 版本匹配 |
| Go | `go list -m -versions <包名>` | 版本在列表中 |

**验证失败处理**：
1. 指定版本不存在 → 降级到最近的可用版本
2. 指定版本已废弃 → 标记为「过时依赖」，建议使用最新稳定版
3. 用户确认后才可继续

**过时检测规则**：
- 检查是否使用了已废弃的API（如 `request` 库在Node.js中已废弃）
- 检查是否使用了项目技术栈声明中未列出的框架版本
- 检查是否使用了与项目已有依赖不兼容的版本

### Step 3：Slopsquatting安全检查

**目标**：防止攻击者抢注AI幻觉包名植入恶意代码。

**检查规则**：

| 检查项 | 说明 | 风险等级 |
|--------|------|----------|
| 包龄检查 | 包创建时间<6个月 | 🟠 新包风险 |
| 下载量检查 | 周下载量<100 | 🟠 低流行度 |
| 维护者检查 | 单一维护者，无其他知名包 | 🟠 可疑维护者 |
| 名称相似度 | 与知名包名高度相似（如 `lodass` vs `lodash`） | 🔴 高度可疑 |
| 依赖关系 | 依赖极少或极多 | 🟠 异常依赖 |

**验证方法**：
- 读取包的npm/pypi页面信息
- 检查包的GitHub仓库是否存在且有活跃度
- 对比包名与知名包的编辑距离

**高风险处理**：
- 标记为「Slopsquatting嫌疑」
- 建议用户手动验证包来源
- 提供替代的知名安全包

## 技术栈一致性检查（P-007）

在Step 1之前，先检查依赖是否与项目技术栈一致：

1. 读取项目的 `package.json` / `pyproject.toml` / `Cargo.toml` 等配置文件
2. 识别项目已使用的技术栈和框架
3. 新依赖是否与现有技术栈兼容？
   - 不兼容 → 标记为「技术栈冲突」，提示用户
   - 兼容 → 继续Step 1

**常见技术栈冲突**：
- 项目用Vue但AI引入React组件
- 项目用Express但AI引入Koa中间件
- 项目用SQLAlchemy但AI引入Django ORM

## 文档幻觉检查（P-008）

在使用依赖的API时，检查API是否真实存在：

1. 读取包的类型定义文件（`.d.ts` / `__init__.py` 等）
2. 检查AI使用的API方法名是否在类型定义中存在
3. 检查参数和返回值是否与类型定义一致

**验证失败处理**：
- API方法不存在 → 搜索正确的API方法名
- 参数不匹配 → 修正为正确的参数签名
- 无法确认 → 标记为「待验证」，建议用户查阅官方文档

## 反模式

- 🚫 跳过验证直接安装依赖（21.7%的包名是AI编造的）
- 🚫 忽略技术栈冲突（项目用Vue却引入React组件）
- 🚫 使用过时废弃的API（如Node.js的`request`库）
- 🚫 忽略Slopsquatting风险（新包+低下载量+名称相似=高危）
- 🚫 不记录验证结果（交付时无法追溯依赖安全性）
