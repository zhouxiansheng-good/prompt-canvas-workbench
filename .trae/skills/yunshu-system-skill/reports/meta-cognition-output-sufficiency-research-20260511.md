# 元认知输出充分性研究报告

日期：2026-05-11

## 研究问题

用户观察到：对 AI 增加“发给我之前，先询问自己是否真的完全掌握和给出所有我要的信息了”后，输出质量明显提升。本报告判断该经验是否有研究依据、是否有人做类似项目，以及如何并入云舒系统每个流程。

## 结论

这条提示大概率有效，但有效点不在“礼貌地让 AI 更努力”，而在于它给模型增加了一个**输出前评价标准**：答没答全、证据够不够、缺口怎么处理、下一步是否清楚。

研究和项目实践支持“自评、反思、验证问题、反馈后修订”可以改善不少任务的输出质量；同时也有研究表明，LLM 在没有外部反馈时并不总能可靠自我纠错，甚至可能让推理更差。因此云舒不应把它设计为“AI 自己想一想就可信”，而应落成：

```text
输出前自检 → 能查先查 → 阻塞再问 → 非阻塞明示假设/限制 → 带证据输出
```

## 研究依据

| 方向 | 代表资料 | 对云舒的启发 |
|------|----------|--------------|
| 自反馈迭代 | [Self-Refine](https://arxiv.org/abs/2303.17651) / [项目页](https://selfrefine.info/) | 输出先被反馈模块检查，再据此修订；云舒可把“给全了吗”作为反馈维度 |
| 验证问题链 | [Chain-of-Verification](https://arxiv.org/abs/2309.11495) | 最终回答前生成并回答验证问题，能降低幻觉；云舒应把“能查先查”作为默认动作 |
| 反思记忆 | [Reflexion](https://arxiv.org/abs/2303.11366) | 从任务反馈中形成反思记忆；云舒可用 context ledger/checkpoint 保存缺口和教训 |
| 元认知提示 | [Metacognitive Prompting](https://arxiv.org/abs/2308.05342) | 系统化自我评估能提升理解类任务；云舒应把自检做成结构化门禁 |
| 自问追问 | [Self-Ask](https://arxiv.org/abs/2210.03350) | 先问并回答子问题，必要时接搜索；云舒可把“用户问题拆成子问题是否答全”加入输出检查 |
| 提示优化 | [OPRO](https://arxiv.org/abs/2309.03409) / [google-deepmind/opro](https://github.com/google-deepmind/opro) | 提示措辞和评价循环会显著影响结果；云舒应沉淀可复用的门禁句 |
| 局限性 | [Large Language Models Cannot Self-Correct Reasoning Yet](https://arxiv.org/abs/2310.01798) | 不应依赖纯内省纠错；关键结论必须接外部证据或工具验证 |
| 工程化工作流 | [obra/superpowers](https://github.com/obra/superpowers) | 已有项目把澄清、计划、TDD、审查、验证做成 agentic skills；云舒可在此基础上加入中文化、检查点、上下文账本和输出充分性 |
| 官方实践边界 | [OpenAI reasoning best practices](https://developers.openai.com/api/docs/guides/reasoning-best-practices) / [Prompt optimizer](https://developers.openai.com/api/docs/guides/prompt-optimizer) | 对新推理模型，简单明确目标和成功标准更重要；提示优化也需要评估集和人工复核 |

## 同类项目判断

有人在做类似方向，但侧重点不同：

- Self-Refine、Reflexion、Chain-of-Verification 更偏研究方法：让模型先反馈、反思或验证再输出。
- OPRO、OpenAI Prompt optimizer 更偏提示自动优化：用评价信号反复改提示。
- Superpowers 更偏软件工程 agent 工作流：通过技能、TDD、审查和证据减少 agent 偏航。
- 云舒的机会是把这些方法合成一个中文研发工作流：调查研究、计划、执行、验收、交付、恢复全部有“输出前是否给全”的门禁。

## 融入云舒的设计

### 全局门禁

每次用户可见输出前执行四问：

1. 用户显式问题是否逐一回答？
2. 用户继续行动所需信息是否给全？
3. 关键结论是否有文件、命令、外部来源或明确假设支撑？
4. 剩余缺口是否已经按“可查证/阻塞/非阻塞”处理？

### 缺口分级

| 类型 | 处理 |
|------|------|
| 可查证缺口 | 先读文件、跑命令、查资料或测试 |
| 阻塞缺口 | 停下，只问一个必要问题 |
| 非阻塞缺口 | 明示假设、限制或后续风险后继续 |

### 各流程嵌入

| 流程 | 输出充分性标准 |
|------|----------------|
| 01-init | 任务卡前确认目标、DoD、约束、非目标足够明确 |
| 02-plan | 计划前确认调研、假设、风险、回滚、验证方式足够 |
| 03-execute | 状态更新前确认改动、证据、影响、下一步说清 |
| 04-accept | 验收通过前确认每条 DoD 有新鲜证据 |
| 05-deliver | 交付前确认用户能复跑、回滚、继续维护 |
| 06-recover | 检查点前确认 phase_memory、checkpoint、context ledger 可恢复 |
| 07-subagent | 分派前确认上下文包自包含；接收结果前确认报告足够控制器判断 |

## 风险与防护

- 风险：自检形式化，变成“全部打勾”的仪式。
  防护：必须标出证据或缺口处理动作。
- 风险：自检导致过度提问，拖慢执行。
  防护：可查证缺口先查，非阻塞缺口明示后继续，只对阻塞缺口提问。
- 风险：AI 相信自己的反思，反而更自信地错。
  防护：关键结论必须接本地验证、外部来源或明确假设，不允许“我觉得”式验收。
