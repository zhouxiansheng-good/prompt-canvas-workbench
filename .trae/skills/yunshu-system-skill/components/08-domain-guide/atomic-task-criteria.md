# 原子任务判定标准

> 定义什么情况下任务"足够简单"，不需要进一步分解。这是"无限细分"的终止条件。

## 核心原则

**"集中优势兵力"** — 任务分解的目的是让每个子任务都简单到LLM可以准确处理，而不是无限拆分。

## 原子任务标准（必须全部满足）

| 标准 | 说明 | 示例 |
|------|------|------|
| **单步可完成** | 任务不需要进一步分解，一步即可完成 | "计算面层厚度" ✓ vs "设计整条道路" ✗ |
| **输入输出明确** | 有清晰的输入和可验证的输出 | "输入：设计速度60km/h，输出：车道数4" ✓ |
| **验收标准可验证** | 完成后可以自动或人工验证 | "运行测试通过" ✓ vs "代码质量良好" ✗ |
| **单一领域** | 不涉及多个领域的知识交叉 | "路面结构设计" ✓ vs "路面+路基+排水综合设计" ✗ |
| **无歧义** | 任务描述清晰，不会产生多种理解 | "计算标准轴载BZZ-100" ✓ vs "确定荷载" ✗ |

## 评估流程

```
输入: 任务描述 + 原子任务标准列表
输出: 是否足够简单（是/否）+ 理由

Step 1: 逐项检查
  对每条标准，判断任务是否满足
  
Step 2: 综合判定
  如果全部满足 → "足够简单，停止分解"
  如果有任何一条不满足 → "需要继续分解"

Step 3: 记录理由
  记录哪条标准不满足，作为分解方向的参考
```

## 动态分解时的评估方法

当进入动态分解模式时，使用以下提示词模板评估任务复杂度：

```
请评估以下任务是否足够简单（不需要进一步分解）：

任务描述：[当前任务描述]

判定标准：
1. 任务可在单步内完成，不需要进一步分解
2. 任务有明确的输入和输出
3. 任务的验收标准可自动验证
4. 任务不涉及多个领域的知识
5. 任务描述清晰，不会产生多种理解

请逐项分析，最后给出结论：
- 如果全部满足：回答"足够简单"，并说明理由
- 如果有不满足的：回答"需要分解"，并指出哪些标准不满足
```

## 反模式

- 🚫 过度分解：把"写一个函数"拆成"定义函数名→写参数→写返回值→写函数体"
- 🚫 分解不足：把"设计整个系统"作为一个原子任务
- 🚫 忽视领域边界：把跨领域的任务强行作为一个原子任务
- 🚫 模糊的验收标准："优化代码"没有明确的完成标准

## 与02-plan的协同

02-plan的WBS分解也遵循原子任务标准：
- 02-plan关注**时间粒度**（≤30分钟）
- 08-domain-guide关注**复杂度粒度**（满足原子任务标准）
- 两者结合：任务既要在30分钟内完成，又要满足原子任务标准
