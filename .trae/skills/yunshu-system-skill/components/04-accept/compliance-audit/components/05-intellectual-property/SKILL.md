# intellectual-property — 知识产权审查组件

> 审查网站项目的知识产权合规性，覆盖开源许可证、第三方依赖冲突、商标使用及用户内容授权条款。

---

## 职责

- 检查开源许可证合规性
- 检查第三方依赖许可证冲突
- 检查商标使用合规性
- 检查用户内容授权条款

---

## 操作流程

### 1. 开源许可证合规审查

1. 扫描项目依赖：
   - 读取 `package.json`、`Cargo.toml`、`requirements.txt`、`go.mod` 等依赖文件
   - 使用 `license-checker` 或类似工具提取所有依赖的许可证信息
2. 检查许可证类型分布：
   - [ ] MIT — 宽松，允许商业使用，需保留版权声明
   - [ ] Apache-2.0 — 宽松，需保留版权声明和许可证文本
   - [ ] BSD — 宽松，需保留版权声明
   - [ ] GPL-2.0/3.0 — 强Copyleft，修改后需开源
   - [ ] LGPL — 弱Copyleft，链接库需开源
   - [ ] AGPL — 最强Copyleft，网络服务也需开源
   - [ ] MPL — 文件级Copyleft
   - [ ] 无许可证 / 自定义许可证 — 需人工审查
   - [ ] 未知许可证 — 需确认
3. 检查许可证文件：
   - [ ] 项目根目录是否有 `LICENSE` 文件
   - [ ] 许可证文件是否与 `package.json` 中声明的一致
   - [ ] 许可证文本是否完整
4. 检查版权声明：
   - [ ] 源代码文件头是否有版权声明
   - [ ] 是否保留所有第三方依赖的版权声明

### 2. 第三方依赖许可证冲突审查

1. 识别许可证兼容性风险：
   - [ ] 项目使用MIT/Apache，但依赖了GPL — 若修改并分发，需GPL开源
   - [ ] 项目闭源商业，但依赖了AGPL — 网络服务也需开源
   - [ ] 项目使用GPL，但依赖了专有许可证 — 可能无法链接
2. 检查依赖链：
   - 间接依赖（transitive dependencies）的许可证
   - 开发依赖（devDependencies）是否混入生产环境
3. 检查分发方式：
   - 前端代码（JS/CSS）分发到用户浏览器 — 属于分发，Copyleft生效
   - 后端代码仅在服务器运行 — AGPL触发，GPL不触发
   - SaaS服务 — AGPL触发开源义务
4. 常见冲突场景：
   | 项目许可证 | 依赖许可证 | 风险 |
   |------------|------------|------|
   | 闭源/商业 | GPL | 分发时需开源 |
   | 闭源/商业 | AGPL | 网络服务也需开源 |
   | MIT | GPL | 分发时整体需GPL |
   | Apache-2.0 | GPL-2.0 | 不兼容，不能组合 |

### 3. 商标使用审查

1. 检查网站使用的商标：
   - [ ] 是否使用了第三方商标（公司Logo、产品名称）
   - [ ] 是否获得了商标使用授权
   - [ ] 是否遵守了商标使用规范（不修改、不暗示背书）
2. 检查开源项目的商标：
   - [ ] 是否遵守了开源项目的商标政策（如React、Vue的商标指南）
   - [ ] 是否在宣传中正确引用技术名称
3. 检查自身商标保护：
   - [ ] 是否有自有商标的注册或申请
   - [ ] 网站名称/Logo是否已查询冲突

### 4. 用户内容授权条款审查

1. 检查用户协议中的授权条款：
   - [ ] 用户上传内容的版权归属（默认归用户）
   - [ ] 平台获得的使用授权范围（展示、存储、修改、再许可）
   - [ ] 授权是否可撤销
   - [ ] 用户删除内容后授权是否终止
2. 检查AI平台特有授权：
   - [ ] 用户提示词是否授权平台用于模型训练
   - [ ] 生成内容是否授权平台展示/分发
   - [ ] 是否允许平台使用生成内容做案例展示
3. 检查UGC平台的授权：
   - [ ] 是否要求用户保证拥有上传内容的版权
   - [ ] 是否声明平台不对用户内容侵权负责
   - [ ] 是否有侵权内容的下架机制
4. 检查开源贡献条款：
   - [ ] 是否要求贡献者签署CLA（贡献者许可协议）
   - [ ] 贡献代码的许可证是否与项目一致

## 输入

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `project_root` | string | 是 | 项目根目录绝对路径 |
| `license_scan_path` | string | 否 | 依赖许可证扫描输出路径 |
| `project_license` | string | 否 | 项目自身许可证类型（默认自动探测） |
| `is_saas` | boolean | 否 | 是否为SaaS服务（影响AGPL判断） |

---

## 输出

| 状态 | 含义 | 后续动作 |
|------|------|----------|
| `IP_OK` | 知识产权审查通过 | 继续下一组件 |
| `IP_WARN` | 存在中低危问题 | 记录风险清单，继续审查 |
| `IP_FAIL` | 存在致命/高危问题 | 记录风险清单，建议阻断 |

## 风险项模板

| 编号 | 严重度 | 问题 | 法规依据 | 修复建议 |
|------|--------|------|----------|----------|
| I001 | 🔴 | 闭源项目依赖AGPL组件 | AGPL第13条 | 移除AGPL依赖或开源项目 |
| I002 | 🔴 | 依赖存在许可证冲突 | 著作权法 | 替换冲突依赖或调整项目许可证 |
| I003 | 🟠 | 未保留第三方版权声明 | MIT/Apache要求 | 补充版权声明和许可证文本 |
| I004 | 🟠 | 用户内容授权范围过宽 | 民法典 | 缩小授权范围，明确可撤销 |
| I005 | 🟡 | 项目缺少LICENSE文件 | 开源最佳实践 | 添加LICENSE文件 |
| I006 | 🟡 | 使用第三方商标未授权 | 商标法 | 获取授权或移除 |
| I007 | 🟢 | 依赖存在未知许可证 | 最佳实践 | 确认许可证或替换依赖 |
| I008 | 🟢 | 未要求贡献者签署CLA | 最佳实践 | 添加CONTRIBUTING.md和CLA |
