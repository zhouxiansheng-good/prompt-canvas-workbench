# 合规审查 — 验收阶段法律风险评估

> **"没有调查就没有发言权"** — 项目上线前的最后一道法律风险闸门。在功能验收通过后、交付前执行，确保项目不因合规缺陷而遭受行政处罚或法律纠纷。

## 触发条件

以下任一条件满足时激活：

| 触发方式 | 具体条件 |
|----------|----------|
| **自动触发** | 04-accept 阶段中，项目被标记为高风险（涉及AI生成、用户数据、支付交易、跨境服务） |
| **用户显式调用** | 用户说"审查合规"/"法律风险"/"合规检查"/"项目风险评估" |
| **门禁触发** | Step 3.5 验收门禁中，审查者认为存在法律合规缺口 |

## 前置条件

- [ ] 项目代码已完成开发（03-execute 结束）
- [ ] 功能验收已通过（04-accept Step 1~3 通过）
- [ ] 部署环境已确定（域名、服务器、目标市场明确）
- [ ] 项目路径可访问

## 输入

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `project_root` | string | 是 | 项目根目录绝对路径 |
| `project_type` | string | 是 | 项目类型：ai-platform / ecommerce / content / saas / other |
| `target_market` | string | 是 | 目标市场：CN / EU / US / JP / 多市场（逗号分隔） |
| `has_payment` | boolean | 否 | 是否包含付费功能（默认false） |
| `has_ugc` | boolean | 否 | 是否包含用户生成内容（默认false） |
| `is_ai_platform` | boolean | 否 | 是否为AI生成内容平台（默认false） |

## 与 04-accept 的集成

合规审查是 **04-accept 验收剧本的可选增强步骤**，在功能验收之后、交付之前执行：

```
04-accept 流程：
  Step 1: 基础验证 → 通过
  Step 2: 规范符合性审查 → 通过
  Step 3: 代码质量审查 → 通过
  Step 3.5: 验收门禁 → 通过
  Step 4: DoD→证据映射 → 完成
  【可选增强】合规审查（本组件）→ 通过/有条件通过/不通过
  Step 5: 交付准备
```

**集成规则**：
- 合规审查 **不替代** Step 1~4 的任何验收步骤
- 合规审查结果 **不影响** 功能验收结论（功能对 ≠ 合法）
- 合规审查结论写入 `acceptance_runbook.md` 的「法律合规」章节
- 致命/高危风险必须修复后方可进入 05-deliver

## 使用流程

```
识别审查范围 → 加载法规指南 → 并行执行五维审查 → 汇总风险清单 → 输出优先级建议 → 写入验收报告
```

### 1. 识别审查范围

确认项目类型、目标市场、技术栈，决定审查重点：

| 项目类型 | 重点审查维度 |
|----------|--------------|
| AI平台 | 内容合规（AI标识）、隐私合规（数据训练）、知识产权（模型授权） |
| 电商 | 交易合规（退款/价格）、隐私合规（订单数据）、安全合规（支付） |
| 内容站 | 内容合规（UGC审核）、隐私合规（Cookie）、知识产权（版权） |
| SaaS | 隐私合规（数据出境）、交易合规（订阅）、知识产权（开源许可） |

### 2. 加载法规指南

读取 `guides/01-china-ai-regulations.md` 获取适用法规条款。

目标市场决定法规集：
- **CN**：个保法、数据安全法、网络安全法、深度合成规定、生成式AI暂行办法
- **EU**：GDPR、DSA、AI Act
- **US**：CCPA/CPRA、COPPA、Section 230
- **JP**：个人信息保护法、特定电信法

### 3. 并行执行五维审查

| 维度 | 组件路径 | 职责 |
|------|----------|------|
| 隐私合规 | `components/01-privacy/SKILL.md` | 隐私政策、数据收集、用户权利、第三方披露 |
| 内容合规 | `components/02-content/SKILL.md` | AI标识、版权、UGC责任、禁止内容过滤 |
| 交易合规 | `components/03-transaction/SKILL.md` | 付费条款、退款、价格、支付安全、税务 |
| 安全合规 | `components/04-security/SKILL.md` | HTTPS、加密、访问控制、漏洞、日志 |
| 知识产权 | `components/05-intellectual-property/SKILL.md` | 开源许可证、依赖冲突、商标、授权条款 |

### 4. 汇总风险清单

合并各维度发现，按严重度分类：

| 等级 | 标识 | 定义 | 处理要求 |
|------|------|------|----------|
| 致命 | 🔴 | 违反强制性法律法规，可能导致行政处罚或刑事责任 | 必须修复，阻断上线 |
| 高危 | 🟠 | 存在重大法律风险或用户权益损害可能 | 必须修复，建议阻断上线 |
| 中危 | 🟡 | 存在合规瑕疵，可能被监管关注或引发纠纷 | 建议修复，记录待办 |
| 低危 | 🟢 | 最佳实践偏离，长期可能影响合规形象 | 可选修复，记录待办 |
| 提示 | ⚪ | 信息提示，供决策参考 | 无需修复，记录备查 |

### 5. 输出优先级建议

- 致命和高危项 → 必须修复后方可进入 05-deliver
- 中危项 → 建议修复，可记录待办后交付
- 低危和提示项 → 记录备查

### 6. 写入验收报告

将合规审查结果追加到 `acceptance_runbook.md`：

```markdown
## 法律合规审查

- 审查日期：YYYY-MM-DD
- 审查范围：隐私/内容/交易/安全/知识产权
- 法规依据：[引用的法规条款]
- 总体评级：[通过/有条件通过/不通过]

### 风险摘要
- 🔴 致命：N项
- 🟠 高危：N项
- 🟡 中危：N项
- 🟢 低危：N项
- ⚪ 提示：N项

### 阻塞项（必须修复）
1. [致命] R001 — ...
2. [高危] R005 — ...

### 修复后复验
修复完成后，重新执行对应维度的审查组件，更新验收报告。
```

## 安全规则（不可覆盖）

1. **不替代律师**：本组件输出仅供参考，不构成法律意见。重大合规决策须咨询执业律师
2. **敏感信息隔离**：审查过程中发现的密钥、凭证、个人隐私数据禁止记录到报告
3. **扫描范围限定**：仅审查项目代码和配置，禁止扫描外部系统或第三方服务内部
4. **结果脱敏**：报告中若引用真实用户数据或密钥，必须脱敏处理
5. **法规时效性**：法规引用以技能内置指南为准，最新法规变动需人工复核

## 错误码

| 错误码 | 含义 | 处理建议 |
|--------|------|----------|
| COM_001 | 项目路径不存在 | 检查 `PROJECT_ROOT` 环境变量 |
| COM_002 | 隐私政策文件未找到 | 确认 `PRIVACY_POLICY_PATH` 或检查项目是否包含隐私政策 |
| COM_003 | 依赖许可证扫描失败 | 检查 `LICENSE_SCAN_PATH` 或手动执行 `license-checker` |
| COM_004 | SSL证书检测失败 | 检查目标域名是否可访问 |
| COM_005 | 法规指南文件缺失 | 确认 `guides/01-china-ai-regulations.md` 存在 |
