## 门禁1：理解与架构门禁

> 解决P-015业务理解缺失/P-016架构判断力缺失/P-017系统级理解不足

**触发条件**：代码库探索完成后自动触发

### A. 系统级依赖分析（P-017）

1. 读取 `safeguards/dependency-analysis.md`
2. 构建3层依赖图谱（L1直接依赖/L2间接依赖/L3系统依赖）
3. 标记高风险节点（枢纽节点/状态节点/边界节点）
4. 生成影响矩阵

### B. 架构决策检查（P-016）

1. 读取 `safeguards/architecture.md`
2. 按6个维度逐项检查（模块拆分/技术选型/数据边界/接口设计/扩展性/部署单元）
3. 检测架构反模式（大泥球/过度设计/技术栈混乱/数据无主/同步地狱）
4. 重要架构决策必须记录ADR

**关键原则**：不构建依赖图谱就不许做任务分解；没有架构决策记录就不许进入Phase 2计划

## 门禁2：90%节点预警门禁

> 解决P-050 90%陷阱

**触发条件**：任务分解完成后自动触发

### 执行动作

1. 读取 `safeguards/vibe-coding.md`，执行90%节点预警
2. 对每个任务评估90%陷阱风险（7类陷阱：边界条件/性能瓶颈/集成问题/用户体验/安全漏洞/数据一致性/遗留兼容）
3. 高风险任务拆分"简单90%"和"困难10%"为独立任务，困难10%标注⚠️预警标签
4. 在计划中明确告知用户哪些任务有90%陷阱风险

**关键原则**：不识别90%陷阱就不许开工，意外卡住比提前预警代价大10倍

## 门禁3：计划质量门禁

> 融合自 Superpowers writing-plans：确保计划文档无占位符、类型一致、规格全覆盖。

**触发条件**：计划自检（2.9）完成后、进入 Phase 3 规范前自动触发

### 执行动作

1. **占位符扫描**：搜索计划文档中的禁止模式
   - "TBD"/"TODO"/"待定"/"后续实现"/"补充细节"
   - "添加适当错误处理"/"添加验证"/"处理边界情况"（无具体代码）
   - "为上述写测试"（无实际测试代码）
   - "类似任务N"（未重复代码）
   - 描述做什么但不展示怎么做
   - 引用未定义的类型/函数/方法

2. **类型一致性扫描**：检查后续任务引用的类型、方法签名、属性名是否与前面任务定义一致

3. **规格覆盖扫描**：逐条检查规格/需求，确认每条都有对应任务实现，列出缺口

### 判定标准

| 检查项 | 通过 | 失败 |
|--------|------|------|
| 占位符扫描 | 无禁止模式 | 发现禁止模式 → 必须修复 |
| 类型一致性 | 名称统一 | 名称不一致 → 必须修复 |
| 规格覆盖 | 每条需求有任务 | 有需求无任务 → 补充任务 |

**关键原则**：计划有占位符或类型不一致 = 计划未完成，不许进入 Phase 3
