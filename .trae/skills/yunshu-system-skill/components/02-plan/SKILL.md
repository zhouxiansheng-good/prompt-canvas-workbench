# 02-plan：主要矛盾分析与计划

> **"在复杂事物的发展过程中，必有一种是主要的矛盾"** — 不抓核心问题就动手，只会越改越乱。

## 职责

输出"可以直接执行"的计划和"决策完备"的规范。核心原则：**先找主要矛盾，再判断阶段，最后才做计划**。

---

## Phase 1：调研与矛盾分析

### 1.0 上下文账本预检

**目标**：复用 01-init 或上一轮计划留下的调查结论，避免重复读取全部材料。

**方法**：

1. 运行 `python scripts/yunshu.py context list --task-id <task_id>`
2. **如无账本**（首次计划或账本丢失）：
   ```bash
   python scripts/yunshu.py context record \
     --task-id <task_id> \
     --phase plan \
     --source "<项目根目录>" \
     --finding "<初始化计划阶段上下文账本>" \
     --action "创建计划阶段初始账本" \
     --gap "<待补充调研>" \
     --next-read "<需读取的代码文件>"
   ```
3. 读取最近且相关的 context：`python scripts/yunshu.py context show <context_id>`
4. 校验新鲜度：`python scripts/yunshu.py context status <context_id>`
5. 对 `fresh` 且覆盖当前计划问题的来源，只使用账本中的压缩结论和关键路径；对 `stale`、`missing`、未覆盖范围，回到原文件补读
6. 将本阶段新增调研、外部资料、取舍和剩余缺口追加到同一任务的 context ledger

**产出**：计划文档中的"调研发现"必须引用 context ledger 的 `context_id`，说明哪些结论来自账本、哪些是本轮新增补读。

**硬规则**：计划阶段结束时必须运行 `python scripts/yunshu.py context record` 刷新账本，否则不许进入 03-execute

### 1.1 代码库探索

**目标**：理解现有架构、找到相关文件、识别已有模式。

**方法**：

| 步骤 | 动作 | 工具 |
|------|------|------|
| 定位 | 找到与任务相关的文件 | 平台搜索工具（优先 rg/grep 等快速文本搜索） |
| 分析 | 读取关键文件，理解实现 | 平台文件读取工具 |
| 模式 | 找到可复用的代码模式 | 平台搜索工具 |
| 架构 | 理清模块间依赖关系 | 平台文件读取工具（入口文件/配置文件） |

**产出**：调研摘要（写入计划文档的"调研发现"章节）

### 1.1.5 理解与架构门禁

> 解决P-015/P-016/P-017。代码库探索完成后自动触发，详见 [gates.md](gates.md) 门禁1。

**关键原则**：不构建依赖图谱就不许做任务分解；没有架构决策记录就不许进入Phase 2计划

### 1.2 主要矛盾分析（5-Why根因分析）

> **"主要矛盾决定事物发展"** — 先找核心问题，再动手。

**目标**：从多个问题中找出核心问题（主要矛盾），区分症状与根因。

**方法**：

1. **列出问题清单**：从用户需求和代码调查中提取所有问题
2. **分析依赖关系**：判断问题之间的因果链
   - 如果A不解决，B就无法解决 → A是B的前置
   - 如果A解决后，B自动消失 → A是B的根因
   - 如果A和B互不影响 → A和B是并列
3. **5-Why分析**：对主要矛盾做5层追问，找到根因

```text
表层症状：xxx
 └─ 为什么？→ xxx
   └─ 为什么？→ xxx
     └─ 为什么？→ xxx
       └─ 为什么？→ [根因]
```

4. **确定主要矛盾**：哪个问题最限制结果、最有传导性、最有决定性

**产出**：主要矛盾 + 根因 + 次要矛盾清单

### 1.3 阶段判断

> **"论持久战"** — 每个阶段有不同的中心任务。

**目标**：判断项目当前阶段，确定架构策略。

**方法**：

| 阶段 | 判断标准 | 架构策略 |
|------|----------|----------|
| 从零搭建 | 无相关代码 | 先搭骨架再填充，最小可用 |
| 功能扩展 | 有基础代码，需加功能 | 开闭原则，向后兼容 |
| 重构优化 | 有代码但结构不合理 | 先加测试再重构，小步重构 |
| Bug修复 | 功能存在但有缺陷 | 根因修复，深度防御 |
| 性能优化 | 功能正确但性能不足 | 瓶颈驱动，数据说话 |

**产出**：当前阶段 + 架构策略 + 沿用/改写/暂停的代码清单

### 1.4 外部知识检索（按需）

**触发条件**：本地代码库无法回答以下问题时
- 使用什么库/框架？
- 当前最佳实践是什么？
- 特定领域的标准做法？

**方法**：联网搜索 → 提取关键信息 → 记录来源

### 1.5 调研自检

- [ ] 我理解了现有架构和相关模块
- [ ] 我找到了可复用的代码模式
- [ ] 我确认了依赖关系和约束
- [ ] 不确定的问题已标记为"待确认"

## Phase 2：计划（Plan）

### 2.0 元认知与输出充分性门禁

> 解决P-053理解债/P-054主观偏差：进入计划阶段前，确认信息完整性。

**触发条件**：Phase 1代码库探索完成后、进入Phase 2之前自动触发

**执行动作**：

1. 读取 `safeguards/meta-cognition.md`
2. 执行五维信息审视（重点维度3上下文 + 维度4假设）
3. 执行输出充分性四问：计划是否有足够调研依据、风险、回滚、验证方式和决策记录
4. 标记所有可查证缺口、🟡非阻塞缺口、🔴阻塞缺口
5. 可查证缺口 → 先补调研；🔴阻塞缺口 → 返回01-init补充信息；🟡非阻塞缺口 → 记录为计划假设或风险后继续
6. 无阻塞缺口 → 进入Phase 2

**关键原则**：上下文不完整时不许做计划，计划基于不完整信息 = 无效计划

### 2.1 目标状态定义

**方法**：从任务卡中提取目标，用一句话描述"完成时系统应该是什么样子"。

```text
目标状态：[一句话描述完成后的系统状态]
当前状态：[一句话描述当前系统状态]
差距：[目标与当前之间的差距]
```

### 2.2 逆向推导

**方法**：从目标状态出发，逆向追问"要达到这个状态，前提是什么？"

```text
目标：用户可以通过 OAuth 登录
  ↑ 前提：OAuth 回调路由已实现
    ↑ 前提：OAuth 配置已写入环境变量
      ↑ 前提：OAuth 库已安装
        ↑ 前提：数据库用户表支持 OAuth 字段
```

### 2.3 文件结构映射

> 融合自 Superpowers writing-plans：在定义任务前，先映射文件职责，锁定分解决策。

**目标**：明确每个文件的创建/修改职责，确保任务分解有据可依。

**方法**：

1. 列出所有需要创建或修改的文件
2. 为每个文件定义单一职责
3. 标注文件间的依赖关系

**映射模板**：

```text
| 文件 | 操作 | 职责 | 依赖 |
|------|------|------|------|
| src/auth/oauth.py | 创建 | OAuth 认证逻辑 | config.py |
| src/auth/routes.py | 修改:15-42 | 认证路由 | oauth.py |
| tests/test_oauth.py | 创建 | OAuth 测试 | oauth.py |
```

**设计原则**：
- 每个文件一个清晰职责
- 优先小而聚焦的文件，避免大而全
- 一起变更的文件放一起，按职责拆分而非按技术层
- 已有代码库遵循既有模式，不单方面重构

### 2.4 任务分解（WBS 工作分解结构）

**原则**：
- **100% 原则**：分解后的任务必须覆盖 100% 的工作范围
- **粒度原则**：每个任务 ≤ 30 分钟可完成，有明确的完成判据
- **依赖原则**：明确标注任务间的前后依赖关系

**分解方法**：

```
Phase 1: [阶段名]
  ├── Task 1.1: [任务名] → 产物: [具体文件/命令]
  ├── Task 1.2: [任务名] → 产物: [具体文件/命令]
  │     └── 依赖: Task 1.1
  └── Task 1.3: [任务名] → 产物: [具体文件/命令]

Phase 2: [阶段名]
  └── ...
```

**递归分解（新增）**：

当任务超过30分钟或不满足原子任务标准时，必须进一步分解：

```
任务分解检查:
  │
  ├─ 任务预计 > 30分钟？
  │   ├─ 是 → 拆分为子任务（每个≤30分钟）
  │   └─ 否 → 检查下一项
  │
  ├─ 任务不满足原子任务标准？（见 components/08-domain-guide/atomic-task-criteria.md）
  │   ├─ 是 → 拆分为更简单的子任务
  │   └─ 否 → 检查下一项
  │
  └─ 任务涉及多个领域？
      ├─ 是 → 按领域拆分为独立子任务
      └─ 否 → 任务粒度合适
```

**递归分解原则**：
- 每个子任务必须有明确的完成判据
- 子任务之间尽量减少依赖（可并行优先）
- 分解深度不超过5层（防止过度分解）
- 分解后重新评估每个子任务的复杂度

**每个任务必须包含**：
- **做什么**：具体操作（创建/修改/删除 哪个文件）
- **为什么**：与目标的关联
- **验证点**：完成后如何确认（编译/测试/检查）
- **风险标记**：是否涉及高风险操作

**细粒度步骤格式**（融合自 Superpowers writing-plans）：

每个任务可进一步拆解为 2-5 分钟的原子步骤，使用 checkbox 语法追踪：

```text
### Task N: [组件名]

**文件**：
- 创建: exact/path/to/file.py
- 修改: exact/path/to/existing.py:123-145
- 测试: tests/exact/path/to/test.py

- [ ] Step 1: 写失败测试
- [ ] Step 2: 运行测试确认失败
- [ ] Step 3: 写最小实现
- [ ] Step 4: 运行测试确认通过
- [ ] Step 5: 提交
```

**步骤粒度原则**：每步是一个原子动作，不是笼统的"实现功能X"。

### 2.3.5 90%节点预警门禁

> 解决P-050 90%陷阱。任务分解完成后自动触发，详见 [gates.md](gates.md) 门禁2。

**关键原则**：不识别90%陷阱就不许开工，意外卡住比提前预警代价大10倍

### 2.3.8 计划质量门禁

> 融合自 Superpowers writing-plans。计划自检完成后自动触发，详见 [gates.md](gates.md) 门禁3。

**关键原则**：计划有占位符或类型不一致 = 计划未完成，不许进入 Phase 3

### 2.4 风险评估

**方法**：对每个识别出的风险，填写风险登记表：

| 风险 | 概率 | 影响 | 应对策略 | 触发条件 | 责任人 |
|------|------|------|----------|----------|--------|
| 依赖升级导致兼容问题 | 中 | 高 | 先在分支测试，确认后合并 | 升级后测试失败 | 用户确认 |
| 数据库迁移不可逆 | 低 | 灾难 | 迁移前备份，写回滚脚本 | 迁移后数据异常 | 用户确认 |

**高风险操作清单**（必须征求用户确认）：
- 🚫 删除文件/目录
- 🚫 依赖升级/降级
- 🚫 数据库 schema 变更
- 🚫 大规模重构（>5 个文件）
- 🚫 运行破坏性命令
- 🚫 修改公共 API 接口

**用户确认交互**（Trae IDE）：
遇到上述高风险操作时，调用 `AskUserQuestion`：
- 问题："以下操作存在 [风险描述]，是否继续？"
- 选项："确认执行" / "取消操作" / "查看回滚方案"
- 用户选择"查看回滚方案" → 展示回滚策略后继续询问

### 2.5 回滚策略

**方法**：为每个高风险操作定义回滚点：

```text
回滚点 1: [操作前状态描述]
  回滚方式: [具体命令/操作]
  回滚验证: [如何确认回滚成功]

回滚点 2: ...
```

**最低要求**：
- 每个高风险操作前必须有一个回滚点
- 回滚方式必须是可执行的命令或步骤
- 回滚后必须有验证方法

### 2.6 执行顺序编排

**方法**：根据依赖关系和风险等级，编排执行顺序：

1. **基础设施先行**：数据库/配置/依赖变更
2. **核心逻辑其次**：业务逻辑实现
3. **集成验证最后**：端到端测试/验收

**产物**：

- `templates/plan.md`（或等价计划文档），包含：
  - 调研发现
  - 目标状态 vs 当前状态
  - 任务分解（WBS）
  - 风险登记表
  - 回滚策略
  - 执行顺序
- Todo 清单（任务拆解，粒度到可验证）

### 2.7 会话分片评估

> 长会话退化预防：在计划阶段就评估是否需要分片，避免执行到一半才退化。

**目标**：判断当前任务是否需要跨多个会话完成，提前规划会话边界。

**评估标准**：

| 指标 | 单会话可完成 | 建议分片 |
|------|-------------|----------|
| 任务数 | ≤5 个 | >5 个 |
| 跨模块数 | ≤2 个模块 | >2 个模块 |
| 预估交互轮次 | ≤20 轮 | >20 轮 |
| 涉及文件数 | ≤10 个 | >10 个 |

**分片策略**：

- 需要分片 → 在WBS中标注会话边界（`--- 会话断点 ---`），每个断点处创建交接文档
- 不需要分片 → 正常执行，但在阶段过渡处仍执行门控检查

**分片标注示例**：

```
Phase 1: [阶段名]
  ├── Task 1.1: ...
  ├── Task 1.2: ...
  --- 会话断点：Phase 1完成后保存上下文 ---
Phase 2: [阶段名]
  ├── Task 2.1: ...
```

### 2.9 计划自检

- [ ] 每个任务都有明确的完成判据
- [ ] 依赖关系已标注，无循环依赖
- [ ] 高风险操作已识别且有回滚策略
- [ ] 非目标（NOT in scope）已明确列出
- [ ] 不确定的问题已标记为"待确认"而非猜测
- [ ] 会话分片评估已完成（是否需要跨会话、断点位置）

**禁止占位符扫描**（融合自 Superpowers writing-plans）：

以下模式是**计划失败**，搜索并修复：
- "TBD"/"TODO"/"待定"/"后续实现"/"补充细节"
- "添加适当错误处理"/"添加验证"/"处理边界情况"（无具体代码）
- "为上述写测试"（无实际测试代码）
- "类似任务N"（必须重复代码，执行者可能乱序阅读）
- 描述做什么但不展示怎么做（代码步骤必须有代码块）
- 引用未在任何任务中定义的类型、函数或方法

**类型一致性检查**（融合自 Superpowers writing-plans）：

- [ ] 后续任务中使用的类型、方法签名、属性名是否与前面任务中定义的一致？
- [ ] 函数名/变量名在所有任务中是否统一？（Task 3 叫 `clearLayers()` 但 Task 7 叫 `clearFullLayers()` 是 Bug）

发现问题 → 内联修复，无需重新审查。发现规格需求无对应任务 → 补充任务。

## Phase 3：规范

### 3.1 输入输出契约

**方法**：对计划中涉及的关键接口/数据结构，定义输入输出：

```text
接口/函数: [名称]
输入: [字段 + 类型 + 约束]
输出: [字段 + 类型 + 约束]
错误: [错误码 + 触发条件 + 返回内容]
```

### 3.2 边界与约束

**方法**：列出硬约束和软约束：

```text
硬约束（不可违反）:
- 性能: [具体指标]
- 安全: [具体要求]
- 兼容性: [具体版本]

软约束（尽量满足）:
- 代码风格: [参考规范]
- 可维护性: [具体要求]
```

### 3.3 测试策略

**方法**：为每个任务定义验证方式：

| 任务 | 验证类型 | 验证命令 | 通过条件 |
|------|----------|----------|----------|
| Task 1.1 | 编译 | `npm run build` | 0 错误 |
| Task 1.2 | 单测 | `npm test` | 全部通过 |
| Task 2.1 | 运行 | 启动服务 | 端口响应 200 |

### 3.4 决策记录

**方法**：把规划过程中的关键决策记录下来，避免执行时反复纠结：

```text
决策 1: [决策内容]
  选项: A) ... B) ...
  选择: [A/B]
  理由: [为什么]
  影响: [对后续步骤的影响]
```

### 3.5 产物

- `templates/spec.md`（或等价规范文档），包含：
  - 输入输出契约
  - 边界与约束
  - 测试策略
  - 决策记录
- `templates/tasks.md`（可选：按模块拆解）

### 3.6 规范自检

- [ ] 所有关键接口的输入输出已定义
- [ ] 硬约束和软约束已区分
- [ ] 每个任务都有对应的验证方式
- [ ] 关键决策已记录理由
- [ ] 规范"决策完备"，执行时不需要临时发明规则

## 路由

计划完成后，根据任务特征选择执行模式：

```
计划 + 规范完成？
  ├─ 否 → 回 01-init
  └─ 是 → 任务特征评估：
       ├─ 任务 ≥3 且基本独立 → `components/07-subagent/SKILL.md`（子智能体驱动）
       ├─ 用户要求 /subagent 或 /parallel → `components/07-subagent/SKILL.md`
       ├─ 任务紧耦合或需调试 → `components/03-execute/SKILL.md`（串行执行）
       └─ 单一任务或简单执行 → `components/03-execute/SKILL.md`
```

**决策铁律**：任务 ≥3 个且基本独立时，优先使用子智能体驱动开发（07-subagent），除非用户明确选择串行执行。

**平台优先规则**：上述路由必须服从当前平台 adapter。Codex 中，任务 ≥3 只是建议，不构成创建子智能体的授权；只有用户明确授权子智能体/并行代理时才进入 07-subagent。
