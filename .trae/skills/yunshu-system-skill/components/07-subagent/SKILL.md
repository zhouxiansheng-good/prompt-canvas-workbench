# 07-subagent：子智能体驱动开发

> **"集中优势兵力，各个歼灭敌人"** — 每个子智能体专注一个任务，隔离上下文，精准执行。
>
> **"统筹全局，协调各方"** — 控制器统筹全局，子智能体各个击破。

## 职责

将计划中的任务分派给独立子智能体执行，每个子智能体拥有隔离的上下文，完成后进行两阶段审查（规范符合性 → 代码质量），确保高质量交付。

## 子组件路由表

| 用户意图 | 子组件路径 | 子组件职责 |
|----------|------------|------------|
| "实现代码" / "写代码" / "开发功能" / "实现任务" | `agents/yunshu-implementer/AGENT.md` | 按云舒微流程实现代码、写测试、自检、提交 |
| "规范审查" / "审查规范" / "检查是否符合规范" | `agents/yunshu-spec-reviewer/AGENT.md` | 验证代码是否严格匹配规范 |
| "质量审查" / "审查质量" / "代码质量检查" | `agents/yunshu-quality-reviewer/AGENT.md` | 验证代码质量、可维护性 |
| "分派任务" / "子智能体" / "并行开发" | `dispatch.md` | 子智能体分派策略、任务拆分、并行执行 |

## 与03-execute的关系

07-subagent 与 03-execute 是**平级的执行模式**，不是附属关系：

```
02-plan 完成后，根据任务特征选择：
  ├─ 任务 ≥3 且基本独立 → 07-subagent（子智能体驱动）
  ├─ 用户要求 /subagent → 07-subagent
  ├─ 任务紧耦合或需调试 → 03-execute（串行执行）
  └─ 单一任务或简单执行 → 03-execute
```

上述路由必须服从当前平台 adapter。Codex 中，只有用户明确授权子智能体/并行代理时才进入本组件；任务 ≥3 只是建议，不构成授权。

| 维度 | 03-execute（串行执行） | 07-subagent（子智能体驱动） |
|------|----------------------|-------------------------|
| 上下文 | 主会话上下文持续累积 | 每个子智能体上下文隔离 |
| 审查 | 代码自检 + 验收阶段审查 | 两阶段审查（规范→质量） |
| 并行 | 串行逐个执行 | 独立任务可并行 |
| 适用 | 紧耦合任务、调试、单一任务 | 多独立任务、大型功能 |
| 成本 | 较低（一个会话） | 较高（每任务1实现者+2审查者） |

## 核心原则

**为什么用子智能体**：
- **上下文隔离**：子智能体不继承主会话的上下文历史，由控制器精确构造所需信息，避免上下文污染
- **专注执行**：每个子智能体只关心一个任务，不被其他任务干扰
- **保留主上下文**：控制器专注于协调工作，不被实现细节淹没
- **独立审查**：审查者不信任实现者的报告，只读代码，比自检更可靠
- **两阶段门禁**：规范审查（不多不少）+ 质量审查（干净可维护），双重保障

**核心公式**：
```
每任务一个子智能体 + 两阶段审查（规范→质量）= 高质量、快迭代
```

**输出充分性原则**：
分派前必须自问“子智能体是否真的拿到了完成任务所需的全部信息？”接收结果前必须自问“实现/审查报告是否足够让我继续做控制器决策？”缺口可查证则补查，阻塞则向子智能体或用户追问，非阻塞则记录为 concerns。

**连续执行原则**：
```
不要在任务之间暂停检查。执行计划中的所有任务，不要中途停下来问"要不要继续"。
唯一需要停下来的理由：BLOCKED 状态无法解决、真正无法推进的歧义、所有任务完成。
```

## 子智能体角色体系

### 角色分工

| 角色 | 智能体名 | 职责 |
|------|-------------|------|
| **控制器**（你） | 云舒系统（主智能体） | 统筹全局、分派任务、审查结果、协调冲突 |
| **实现者** | `yunshu-implementer` | 按云舒微流程实现代码、写测试、自检、提交 |
| **规范审查者** | `yunshu-spec-reviewer` | 按云舒微流程验证代码是否严格匹配规范 |
| **质量审查者** | `yunshu-quality-reviewer` | 按云舒微流程验证代码质量 |

### 智能体前置条件

使用子智能体驱动开发前，必须确认当前平台支持子智能体，并读取对应 adapter：

- Trae IDE：按 `adapters/trae/agent-install.md` 创建以下 3 个智能体。
- Codex：只有用户明确授权子智能体/并行代理时才启用，按 Codex adapter 映射。
- 其他平台：先确认是否有等价 subagent / task tool；没有则回退到 `03-execute` 串行执行。

1. **yunshu-implementer** — 实现者智能体
2. **yunshu-spec-reviewer** — 规范审查者智能体
3. **yunshu-quality-reviewer** — 质量审查者智能体

智能体创建提示词见 `agents/` 目录下的 AGENT.md 文件。

## 执行流程

### Step 1：加载计划与任务提取

1. 读取计划文件（plan.md + spec.md + tasks.md）
2. **一次性提取所有任务的完整文本和上下文**（后续不再重复读取文件）
3. 识别任务间依赖关系，确定执行顺序
4. 识别可并行的任务组
5. 使用当前平台的计划/任务跟踪能力列出所有任务；无专用工具时维护一份可见任务清单

**任务分类**：

| 类型 | 特征 | 执行方式 |
|------|------|---------|
| **串行任务** | 后续任务依赖前序任务的产物 | 逐个分派 |
| **并行任务** | 无共享状态、无依赖关系 | 同时分派 |
| **条件任务** | 依赖前序任务的结果决策 | 前序完成后评估 |

### Step 1.5：Agent安全门禁

> 分派子智能体前必须执行的安全检查，防止Tool Poisoning/Rug Pull/无限循环等Agent安全风险。

**触发条件**：进入07-subagent组件时自动触发（本组件专门处理Agent分派）

**执行动作**：

1. 读取 `safeguards/agent-safety.md`
2. 逐项检查7类Agent安全风险（Tool Poisoning→Rug Pull→信任链→无限循环→上下文腐败→成本控制→非确定性）
3. 发现🔴致命风险（Tool Poisoning/Rug Pull）→ 停止分派，修复后重试
4. 发现🟠严重风险（无限循环/上下文腐败/成本爆炸/非确定性）→ 记录风险，加强监控后继续

**分派安全约束**（所有子智能体分派时必须遵守）：

| 约束 | 限制值 | 说明 |
|------|--------|------|
| 单任务工具调用上限 | ≤10次 | 超过5次请求用户确认 |
| 重复调用检测 | 连续3次相同调用→强制停止 | 防止无限循环 |
| 并行Agent数量 | ≤3个 | 防止成本爆炸 |
| 关键信息持久化 | 必须写入文件 | 不只依赖上下文传递 |
| 工具来源验证 | 只用官方或已验证工具 | 防止Tool Poisoning |
| 子智能体输出验证 | 必须通过两阶段审查 | 防止非确定性风险 |

### Step 1.6：分派前输出充分性门禁

读取 `safeguards/meta-cognition.md`，执行输出充分性四问，重点检查：

- 分派包是否包含任务目标、范围、非目标、约束、验收标准、工作目录？
- 是否提供了足够代码上下文、依赖关系和已有模式，而不是让子智能体从零猜？
- 文件写入范围是否明确，是否避免多个子智能体修改同一文件？
- 预期返回格式是否包含 status、files_changed、evidence、concerns、output_sufficiency？

任一阻塞缺口未补齐 → 不许分派。

### Step 2：逐任务分派子智能体

**每个任务的完整流程**：

```
分派 yunshu-implementer 智能体
  ↓
实现者提问？→ 回答问题，提供上下文 → 重新分派
  ↓ 否
实现者完成实现、测试、提交、自检
  ↓
分派 yunshu-spec-reviewer 智能体
  ↓
规范审查通过？
  ├─ 否 → yunshu-implementer 修复规范偏差 → 重新审查
  └─ 是 → 分派 yunshu-quality-reviewer 智能体
              ↓
         质量审查通过？
           ├─ 否 → yunshu-implementer 修复质量问题 → 重新审查
           └─ 是 → 标记任务完成 → 下一个任务
```

**铁律**：
```
规范审查未通过 → 不许进入质量审查
质量审查未通过 → 不许标记完成
审查发现问题 → 必须修复后重新审查（不许跳过复审）
不要在任务之间暂停问用户"要不要继续"
```

### Step 3：分派实现者智能体

调用 `yunshu-implementer` 智能体，传递：

> 子智能体已内置云舒微流程（澄清→计划→执行→自验收），控制器只需提供完整上下文，子智能体会自行按微流程执行。

**平台调用映射**：

- Trae IDE：使用 Task tool / 智能体面板，智能体名 `yunshu-implementer`。
- Codex：仅在用户明确授权时使用 Codex subagent，并说明该 subagent 的职责和文件范围。
- 其他平台：使用对应 adapter 中的 subagent 调用方式；没有等价能力则切回串行执行。

```text
<当前平台的子智能体调用方式>
agent/subagent: "yunshu-implementer"
  description: "实现任务 N：[任务名]"
  query: |
    任务描述：[完整粘贴任务文本]
    上下文：[任务在整体中的位置、依赖、架构上下文、已有代码模式]
    工作目录：[目录路径]
    [可选] 领域上下文包：[来自 08-domain-guide 的 context_package，如果存在]
    [可选] 软件技能：[context_package.software_skills 列表，如果存在]
```

**领域上下文包注入**：

如果任务来自经过 08-domain-guide 领域引导的流程，必须在分派时注入领域上下文包：

```
分派 yunshu-implementer 智能体
  query: |
    任务描述：[任务文本]
    领域上下文包：
      domain_id: [领域ID]
      selections: [用户选择序列]
      context: [注入的上下文参数]
      constraints: [领域约束条件]
    软件技能：[可用的软件技能ID列表]
    ...
```

**子智能体使用领域上下文的规则**：
- 实现者必须将领域上下文包中的约束条件纳入计划和实现
- 实现者应根据 software_skills 列表选择合适的工具/库
- 实现者应参考 context 中的适用规范和设计参数

### Step 4：处理实现者状态

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| **DONE** | 完成，无异议 | 进入规范审查 |
| **DONE_WITH_CONCERNS** | 完成但有疑虑 | 读取疑虑：正确性/范围问题 → 先解决再审查；观察性意见 → 记录并继续审查 |
| **NEEDS_CONTEXT** | 缺少信息 | 提供缺失上下文，重新分派 |
| **BLOCKED** | 无法完成 | 评估阻塞：上下文问题→补充后重派；任务过大→拆分；计划有误→升级给用户 |
| **DECOMPOSITION_REQUEST** | 请求分解 | 评估分解建议 → 确认分解或拒绝 |

**铁律**：
```
永远不要忽略子智能体的升级请求
子智能体说卡住了 → 一定有什么需要改变
```

**处理 DECOMPOSITION_REQUEST**：

```
收到 DECOMPOSITION_REQUEST:
  │
  ├─ 评估分解建议
  │   ├─ 子任务是否明确？
  │   ├─ 每个子任务是否满足原子任务标准？
  │   ├─ 子任务之间依赖是否合理？
  │   └─ 分解后是否能覆盖原任务范围？
  │
  ├─ 确认分解
  │   ├─ 创建子任务列表
  │   ├─ 更新任务依赖关系
  │   ├─ 重新分派第一个子任务
  │   └─ 原任务标记为"已分解"
  │
  └─ 拒绝分解（如果评估不通过）
      ├─ 向子智能体说明拒绝理由
      ├─ 要求子智能体继续执行原任务
      └─ 或升级给用户决策
```

**分解协同规则**：
- 子智能体分解请求与 08-domain-guide 动态分解协同
- 如果任务来自领域引导流程，优先使用 08-domain-guide 的动态分解
- 子智能体分解作为补充机制，处理计划阶段未预见到的复杂度

### Step 5：分派规范审查者智能体

调用 `yunshu-spec-reviewer` 智能体，传递：

> 子智能体已内置云舒微流程（澄清→计划→审查→自验收），控制器只需提供完整上下文，子智能体会自行按微流程执行。

```text
<当前平台的子智能体调用方式>
agent/subagent: "yunshu-spec-reviewer"
  description: "审查任务 N 的规范符合性"
  query: |
    规范要求：[完整粘贴任务规范]
    实现者报告：[来自实现者的报告]
    变更文件：[实现者修改的文件列表]
```

**规范审查铁律**：
- 规范审查者发现的问题 = 未完成，不是"差不多"
- 实现者修复后必须重新审查
- 不许跳过复审
- **实现者的自检不能替代正式审查**（两者都需要）

### Step 6：分派质量审查者智能体

调用 `yunshu-quality-reviewer` 智能体，传递：

> 子智能体已内置云舒微流程（澄清→计划→审查→自验收），控制器只需提供完整上下文，子智能体会自行按微流程执行。

```text
<当前平台的子智能体调用方式>
agent/subagent: "yunshu-quality-reviewer"
  description: "审查任务 N 的代码质量"
  query: |
    任务概要：[来自实现者的报告摘要]
    规范要求：[任务 N 来自计划文件]
    变更范围：BASE_SHA [任务前提交] → HEAD_SHA [当前提交]
```

**质量审查铁律**：
- 只在规范审查通过后才分派质量审查
- 质量审查发现问题 → 实现者修复 → 重新质量审查
- 不许跳过复审

### Step 7：并行任务分派

当多个任务完全独立（无共享状态、无依赖关系）时，可并行分派：

**并行分派条件**：

| 条件 | 说明 |
|------|------|
| 任务间无依赖 | 后续任务不依赖前序任务的产物 |
| 无共享文件 | 不同任务不修改同一文件 |
| 无共享状态 | 不同任务不操作同一资源 |

**并行分派流程**：

```
1. 识别可并行的任务组
2. 同时分派多个 yunshu-implementer 智能体
3. 等待所有智能体完成
4. 逐个进行规范审查 + 质量审查
5. 检查并行结果是否冲突
6. 运行全量测试确认集成
```

**并行分派铁律**：
```
并行只限实现者，审查必须串行
并行任务不能修改同一文件
并行完成后必须检查冲突
```

**并行分派的提示词结构**：

好的并行分派提示词是：
1. **聚焦** — 一个清晰的问题域
2. **自包含** — 理解问题所需的所有上下文
3. **明确输出** — 子智能体应该返回什么

```
❌ 太宽泛："修复所有测试" — 子智能体会迷失
✅ 具体的："修复 agent-tool-abort.test.ts 中的 3 个失败测试"

❌ 无上下文："修复竞态条件" — 子智能体不知道在哪
✅ 有上下文：粘贴错误信息和测试名称

❌ 无约束：子智能体可能重构一切
✅ 有约束："不要修改生产代码" 或 "只修测试"

❌ 模糊输出："修复它" — 你不知道改了什么
✅ 明确输出："返回根因和变更的摘要"
```

### Step 8：全部任务完成后 + 阶段留底

1. 分派 yunshu-quality-reviewer 做最终全量审查（审查整体实现的一致性和集成性）
2. 运行全量测试套件
3. **创建子智能体阶段检查点**（强制）：
   ```bash
   python scripts/yunshu.py checkpoint create \
     --task-id <task_id> \
     --phase subagent \
     --summary "<子智能体执行阶段完成摘要>" \
     --completed "<全部已完成子任务列表>" \
     --pending "<待验收事项>" \
     --artifact "<关键产物路径>" \
     --next-action "进入04-accept正式验收"
   python scripts/yunshu.py validate checkpoint .yunshu/checkpoints/<checkpoint_id>/checkpoint.json
   ```
4. **更新上下文账本**（强制）：
   ```bash
   python scripts/yunshu.py context record \
     --task-id <task_id> \
     --phase subagent \
     --source "<子智能体修改的文件列表>" \
     --finding "<子智能体执行阶段关键发现>" \
     --action "<已完成子任务和审查>" \
     --gap "<遗留问题>" \
     --next-read "<验收阶段需读取的文件>"
   ```
5. 进入 `components/04-accept/SKILL.md` 正式验收

**硬规则**：没有创建阶段检查点 → 不许进入 04-accept

## 智能体调用决策

**何时启用子智能体模式**：当前平台 adapter 允许且任务 ≥3 且独立 / 用户要求 / 大型功能 / 多文件协作 / 并行加速

**三大智能体调用顺序**：implementer → spec-reviewer → quality-reviewer（顺序不能反）

**快速参考**：

| 你现在想做什么？ | 调用哪个智能体 |
|----------------|--------------|
| 开始实现一个新任务 | `yunshu-implementer` |
| 实现完成了，检查对不对 | `yunshu-spec-reviewer` |
| 规范对了，检查代码好不好 | `yunshu-quality-reviewer` |
| 所有任务做完，整体检查一遍 | `yunshu-quality-reviewer`（最终全量审查） |
| 审查说有问题，要修复 | `yunshu-implementer`（修复后重新审查） |

→ 详见 [dispatch.md](dispatch.md)：三大智能体调用时机铁律、完整调用流程图、并行调用场景、调用前置检查

- 🚫 让子智能体读计划文件（应提供完整文本，不浪费上下文）
- 🚫 跳过规范审查直接做质量审查（顺序不能反）
- 🚫 审查发现问题但不修复就继续（必须修复后复审）
- 🚫 用实现者的自检替代正式审查（两者都需要）
- 🚫 并行分派修改同一文件的子智能体（会冲突）
- 🚫 忽略子智能体的提问或升级请求（必须回应）
- 🚫 接受"差不多"的规范符合（规范审查发现问题 = 未完成）
- 🚫 在规范审查通过前开始质量审查（顺序错误）
- 🚫 子智能体间共享上下文（每个子智能体应有隔离上下文）
- 🚫 在任务之间暂停问用户"要不要继续"（直接执行所有任务）
- 🚫 让子智能体自己读文件获取上下文（控制器应直接提供完整文本）
- 🚫 不审查工具描述就分派子智能体（Tool Poisoning风险）
- 🚫 不设工具调用次数上限（无限循环风险）
- 🚫 不验证子智能体输出就标记完成（非确定性风险）
- 🚫 同时运行超过3个子智能体（成本爆炸风险）
- 🚫 关键决策只存在上下文中不写入文件（上下文腐败风险）
- 🚫 远程工具授权后不锁定版本（Rug Pull风险）
