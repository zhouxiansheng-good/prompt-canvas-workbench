# 智能体调用决策

## 何时进入子智能体模式

```
02-plan 制定计划完成 → 评估是否使用子智能体
    ↓
┌─────────────────────────────────────────────┐
│  判断条件（满足任一即启用子智能体模式）：    │
│                                             │
│  ✅ 任务数量 ≥ 3 且基本独立                  │
│  ✅ 用户明确说 "用/子智能体" 或 "/subagent"   │
│  ✅ 大型功能开发（预计 > 500 行代码变更）       │
│  ✅ 需要多文件协作实现                         │
│  ✅ 用户要求并行加速                          │
│                                             │
│  否则 → 使用 03-execute 串行执行              │
└─────────────────────────────────────────────┘
```

## 三大智能体的调用时机

### yunshu-implementer（实现者）

| 触发场景 | 调用条件 | 示例 |
|---------|---------|------|
| **常规分派** | 计划中的每个待执行任务 | "实现用户认证模块" |
| **修复规范偏差** | 规范审查发现问题后 | "审查发现缺了密码验证，请补上" |
| **修复质量问题** | 质量审查发现问题后 | "审查发现有重复代码，请提取" |
| **重新分派** | 实现者返回 NEEDS_CONTEXT | 补充上下文后重新分派 |
| **拆分重派** | 实现者返回 BLOCKED（任务过大） | 拆分为子任务后重新分派 |

**调用前置检查**：
- [ ] 任务描述已完整写入 query
- [ ] 相关上下文（架构、依赖、代码模式）已提供
- [ ] 工作目录路径正确
- [ ] 任务间的依赖关系已明确

**不调用的场景**：
- ❌ 单行修改或简单配置更改（直接做更快）
- ❌ 需要调试已有 bug（用 03-execute 的系统化调试）
- ❌ 需要与用户频繁交互澄清需求（先沟通再分派）

### yunshu-spec-reviewer（规范审查者）

| 触发场景 | 调用条件 | 前置状态 |
|---------|---------|---------|
| **首次审查** | 实现者返回 DONE 或 DONE_WITH_CONCERNS | 实现必须完成 |
| **复审** | 实现者修复规范偏差后 | 必须重新审查 |
| **最终确认** | 用户质疑实现是否符合规范 | 独立验证 |

**调用时机铁律**：

```
⛔ 绝不在以下情况调用规范审查者：
   - 实现者还未完成实现
   - 实现者返回 NEEDS_CONTEXT 或 BLOCKED
   - 还未看过实现者的报告和代码变更

✅ 必须在以下情况调用规范审查者：
   - 实现者声明完成后（DONE / DONE_WITH_CONCERNS）
   - 已获取实现者的完整报告
   - 已知道变更的文件列表
```

**审查重点（告诉规范审查者看什么）**：
- 是否实现了规范要求的**所有功能**（不多不少）
- 是否遗漏了规范中的**任何细节**
- 是否实现了规范中**没有要求的功能**（范围蔓延）
- 边界情况和错误处理是否符合规范

### yunshu-quality-reviewer（质量审查者）

| 触发场景 | 调用条件 | 前置状态 |
|---------|---------|---------|
| **单任务质量审查** | 规范审查通过后 | 规范审查必须通过 ✅ |
| **复审** | 实现者修复质量问题后 | 必须重新审查 |
| **最终全量审查** | 所有任务完成后 | 整体验收前 |

**调用时机铁律**：

```
⛔ 绝不在以下情况调用质量审查者：
   - 规范审查还未通过（顺序不能反！）
   - 实现者还在修复规范偏差
   - 还未获取规范审查通过的确认

✅ 必须在以下情况调用质量审查者：
   - 规范审查者明确表示 "通过" 或无阻塞问题
   - 已获取实现者的报告摘要
   - 已知道变更范围（BASE_SHA → HEAD_SHA）
```

**审查重点（告诉质量审查者看什么）**：
- 代码可读性和命名清晰度
- 是否有重复代码或坏味道
- 测试覆盖率和测试质量
- 文件组织结构是否合理
- 错误处理是否完善

## 并行调用场景

当多个任务**完全独立**时，可以并行调用多个 implementer：

```
独立任务组 [Task A, Task B, Task C]
         │
         ▼
┌─────────────────────────────────────────┐
│  同时调用：                               │
│  • implementer 实现 Task A               │
│  • implementer 实现 Task B               │
│  • implementer 实现 Task C               │
└─────────────────────────────────────────┘
         │
         ▼ (全部实现完成)
┌─────────────────────────────────────────┐
│  串行审查（必须逐个来）：                  │
│  1. spec-reviewer 审查 Task A           │
│  2. quality-reviewer 审查 Task A         │
│  3. spec-reviewer 审查 Task B           │
│  4. quality-reviewer 审查 Task B         │
│  5. spec-reviewer 审查 Task C           │
│  6. quality-reviewer 审查 Task C         │
└─────────────────────────────────────────┘
         │
         ▼
   检查冲突 → 运行全量测试
```

**并行规则**：
- ✅ 可以并行：多个 implementer 同时实现不同任务
- ⛔ 不可以并行：审查者必须串行（避免上下文混乱）
- ⛔ 不可以并行：修改同一文件的任务

## 快速参考卡片

| 你现在想做什么？ | 调用哪个智能体 |
|----------------|--------------|
| 开始实现一个新任务 | `yunshu-implementer` |
| 实现完成了，检查对不对 | `yunshu-spec-reviewer` |
| 规范对了，检查代码好不好 | `yunshu-quality-reviewer` |
| 所有任务做完，整体检查一遍 | `yunshu-quality-reviewer`（最终全量审查） |
| 审查说有问题，要修复 | `yunshu-implementer`（修复后重新审查） |
