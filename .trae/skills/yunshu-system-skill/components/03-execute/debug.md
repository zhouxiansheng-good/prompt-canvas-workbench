# 系统化调试

**铁律**：
```
没有根因调查 → 不许提修复方案
```

## 何时使用

用于任何技术问题：测试失败、生产环境 Bug、意外行为、性能问题、构建失败、集成问题。

**特别在以下情况使用**：
- 时间紧迫时（紧急情况让猜测变得诱人）
- "只是一个快速修复"看起来很明显
- 你已经尝试了多个修复
- 之前的修复没有生效
- 你不完全理解问题

**不要跳过**：
- 问题看起来简单（简单的 Bug 也有根因）
- 你很着急（匆忙保证返工）
- 经理要求现在就修（系统化比瞎折腾快）

## CI 失败快速入口

CI 检查失败时，可直接使用以下流程定位根因：

```
1. 识别失败检查：gh pr checks --json name,bucket,state,workflow,link
2. 获取失败日志：gh run view <run-id> --log-failed
3. 外部检查：访问提供的链接提取失败信息
4. 定位根因 → 最小化修复 → 推送验证
```

**边界条件**：
- 每次修复限定在单一失败原因
- 不绕过钩子（`--no-verify`）强制推送
- 失败与 PR 无关时，合并最新 main 而非膨胀 PR
- 抖动失败重试一次并报告证据
- 每次推送后重新运行 `gh pr checks`，检查集可能变化

## Phase 0：查询 Bug 知识库

**铁律**：
```
先查知识库 → 再开始调试
```

**查询流程**：

1. **定位知识库目录**
   - 默认路径：`.yunshu/bug-knowledge/`
   - 如果目录不存在 → 跳过此步骤，继续 Phase 1

2. **提取搜索关键词**
   - 从错误信息中提取关键词
   - 从症状描述中提取关键词
   - 从代码上下文中提取关键词

3. **搜索匹配的 Bug 记录**
   ```bash
   grep -l "关键词1\|关键词2" .yunshu/bug-knowledge/*.md
   grep -l "type: logic" .yunshu/bug-knowledge/*.md
   grep -l "症状描述关键词" .yunshu/bug-knowledge/*.md
   ```

4. **评估匹配结果**
   - 阅读匹配的 Bug 知识卡
   - 对比症状是否相似
   - 对比根因是否相关
   - 对比代码模式是否匹配

5. **决策分支**

   | 匹配程度 | 行动 |
   |----------|------|
   | **完全匹配** | 直接应用历史修复方案，跳到 Phase 4 |
   | **部分匹配** | 参考历史方案，继续 Phase 1-3 |
   | **无匹配** | 继续 Phase 1，从头开始调试 |

6. **记录查询结果**
   - 在调试会话记录中记录查询结果
   - 如果找到匹配，记录引用的 Bug ID

**反模式**：
- 🚫 跳过知识库查询直接调试 — 可能重复踩坑
- 🚫 忽略匹配结果 — 浪费历史经验
- 🚫 盲目应用不匹配的方案 — 可能引入新问题

## Phase 1：根因调查

**在尝试任何修复之前**：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们经常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致性复现**
   - 能可靠触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变了可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录什么数据进入组件
     - 记录什么数据离开组件
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次收集证据，显示在哪里断裂
   然后分析证据识别失败组件
   然后调查那个特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的密钥： ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量： ==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== 钥匙串状态： ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

5. **追踪数据流**

   **向后追踪技术**：
   - 错误值从哪里来？
   - 谁用错误值调用了这个？
   - 继续往上追溯直到找到源头
   - 在源头修复，不是在症状处

   **添加堆栈追踪**：
   ```typescript
   async function gitInit(directory: string) {
     const stack = new Error().stack;
     console.error('DEBUG git init:', {
       directory,
       cwd: process.cwd(),
       nodeEnv: process.env.NODE_ENV,
       stack,
     });
     await execFileAsync('git', ['init'], { cwd: directory });
   }
   ```

   **运行并捕获：**
   ```bash
   npm test 2>&1 | grep 'DEBUG git init'
   ```

## Phase 2：模式分析

**在修复之前找到模式**：

1. **找到工作的例子**
   - 在同一代码库中定位类似的正常工作代码
   - 什么工作的与什么损坏的类似？

2. **与参考对比**
   - 如果实现模式，完整阅读参考实现
   - 不要略读 - 阅读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作的和损坏的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不可能有影响"

4. **理解依赖**
   - 这需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做了什么假设？

---

## Phase 3：假设与测试

**科学方法**：

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的改动来测试假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **验证后再继续**
   - 有效？是 → Phase 4
   - 没效？形成新假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 更多研究

## Phase 4：实施修复

**修复根因，不是症状**：

1. **创建失败测试用例**
   - 最简单的复现
   - 尽可能自动化测试
   - 如果没有框架，用一次性测试脚本
   - 修复前必须有

2. **实施单一修复**
   - 解决识别的根因
   - 一次一个改动
   - 没有"既然来了"的改进
   - 没有捆绑重构

3. **增量验证修复法**

   ```
   修改前：记录基线（当前测试结果）
     ↓
   步骤 1：做最小改动
     ↓
   立即验证：跑测试，确认步骤 1 无副作用
     ↓
   步骤 2：做下一个最小改动
     ↓
   立即验证：跑测试，确认步骤 2 无副作用
     ↓
   ... 重复直到修复完成
     ↓
   全量回归验证
   ```

   **增量验证规则**：
   - 每步修改 ≤ 1 个函数/1 个文件
   - 每步修改后立即运行相关测试
   - 测试失败 → 立即回退，分析原因
   - 不在失败的修改上叠加新修改

4. **验证修复**
   - 测试现在通过？
   - 没有其他测试损坏？
   - 问题实际解决？

5. **深度防御**

   **四层防御模型**：

   | 层级 | 目的 | 示例 |
   |------|------|------|
   | **第 1 层：入口验证** | 在 API 边界拒绝明显无效输入 | 函数参数校验：空值/类型/范围 |
   | **第 2 层：业务逻辑验证** | 确保数据对当前操作有意义 | 操作前状态检查 |
   | **第 3 层：环境守卫** | 在特定上下文中阻止危险操作 | 测试环境拒绝生产操作 |
   | **第 4 层：调试工具** | 为取证捕获上下文 | 关键操作前记录堆栈和参数 |

   **应用模式**：
   ```
   找到 Bug 后：
   1. 追踪数据流 — 错误值从哪来？经过哪些层？
   2. 映射所有检查点 — 数据经过的每一个点
   3. 在每层添加验证 — 入口、业务、环境、调试
   4. 测试每层 — 尝试绕过第 1 层，验证第 2 层能捕获
   ```

6. **回归验证**

   **铁律**：
   ```
   修复后不跑回归测试 = 没有完成修复
   ```

   **回归验证流程**：
   ```
   修复完成
     ↓
   运行原始测试（确认 Bug 已修）
     ↓
   运行全量测试套件（确认无回归）
     ↓
   检查影响矩阵中的所有调用方
     ↓
   针对受影响的调用方运行专项测试
     ↓
   全部通过 → 修复确认
   任一失败 → 回退修改，重新分析
   ```

   **回归测试范围确定**：

   | 修改类型 | 回归范围 |
   |----------|----------|
   | 私有函数内部修改 | 该模块的单元测试 |
   | 公共 API 参数变更 | 所有调用方 + 全量测试 |
   | 共享工具函数修改 | 全量测试 |
   | 配置/环境变更 | 集成测试 + 全量测试 |
   | 数据库/数据结构变更 | 全量测试 + 迁移测试 |

7. **同类型 Bug 批量修复**

   **铁律**：
   ```
   修复一个 Bug → 必须搜索同类 Bug → 批量修复
   ```

   **批量修复流程**：
   ```
   单个 Bug 修复完成
     ↓
   提取 Bug 代码模式（用于搜索的特征）
     ↓
   全局搜索相同模式
     ↓
   逐个评估是否需要修复
     ↓
   批量应用修复（增量验证）
     ↓
   全量回归验证
   ```

   **步骤 1：提取 Bug 代码模式**

   | 特征类型 | 示例 |
   |----------|------|
   | **代码模式** | `if (x = null)` → 应为 `if (x == null)` |
   | **API 误用** | `array.push()` → 应为 `array.append()` |
   | **边界条件** | `for (i = 0; i <= len; i++)` → 应为 `i < len` |
   | **类型错误** | `parseInt(x)` → 应为 `parseInt(x, 10)` |
   | **资源泄漏** | `open(file)` 无 `close()` |

   **步骤 2：全局搜索相同模式**
   ```bash
   grep -rn "if (. = null)" --include="*.js" src/
   grep -rn "\.push\(" --include="*.py" src/
   grep -rn -E "pattern" --include="*.ext" path/
   ```

   **步骤 3：逐个评估**

   | 风险等级 | 条件 | 行动 |
   |----------|------|------|
   | 高 | 确定是同类 Bug | 必须修复 |
   | 中 | 可能是同类 Bug | 需要验证 |
   | 低 | 不太可能是 Bug | 记录观察 |

   **步骤 4：批量应用修复**（增量验证：修复一个 → 验证一个 → 再修复下一个）

   **步骤 5：全量回归验证**

   **反模式**：
   - 🚫 修复一个 Bug 就结束 — 可能遗漏同类 Bug
   - 🚫 一次性修改所有文件 — 无法定位问题
   - 🚫 跳过增量验证 — 可能引入新问题
   - 🚫 不做回归测试 — 无法确认批量修复的安全性

8. **存入 Bug 知识库**

   **铁律**：
   ```
   修复完成 → 必须存入知识库
   ```

   **存储流程**：

   1. 创建知识库目录：`mkdir -p .yunshu/bug-knowledge`
   2. 生成 Bug ID：`YYYY-MM-DD-<type>-<short-hash>`（类型：logic / null-pointer / boundary / concurrency / type-mismatch / config / dependency / other）
   3. 使用模板创建知识卡：模板 `templates/bug_knowledge.md`，输出 `.yunshu/bug-knowledge/<bug-id>.md`
   4. 填写知识卡内容（必填：症状描述、根因分析、修复方法、代码模式、影响范围、验证方法；可选：预防措施、相关 Bug、教训总结）
   5. 关联同类 Bug
   6. 验证知识卡完整性

   **反模式**：
   - 🚫 修复完成不存知识库 — 浪费经验
   - 🚫 知识卡内容不完整 — 无法复用
   - 🚫 代码模式不可搜索 — 无法找到同类 Bug
   - 🚫 不关联同类 Bug — 知识碎片化

9. **如果修复无效**
   - 停止
   - 计数：你尝试了多少次修复？
   - 如果 < 3：回到 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（步骤 10）**

10. **如果 3+ 次修复失败：质疑架构**

    **指示架构问题的模式：**
    - 每次修复在不同地方揭示新的共享状态/耦合/问题
    - 修复需要"大规模重构"才能实施
    - 每次修复在其他地方制造新症状

    **停止并质疑基础：**
    - 这个模式根本上合理吗？
    - 我们是否"仅凭惯性坚持"？
    - 我们应该重构架构还是继续修复症状？

    **在尝试更多修复之前与用户讨论**

## 自纠正循环机制

当验证失败时，启动自纠正循环：

```
验证失败
  ↓
Phase 0：查询 Bug 知识库
  ↓
进入调试流程（Phase 1-4）
  ↓
变更前影响分析（2.1）
  ↓
增量验证修复（Phase 4.3）
  ↓
深度防御加固（Phase 4.5）
  ↓
回归验证（Phase 4.6）
  ↓
同类型 Bug 批量修复（Phase 4.7）
  ↓
存入 Bug 知识库（Phase 4.8）
  ↓
通过？→ 继续
失败？→ 循环计数 +1
  ↓
循环计数 < 3？→ 回到 Phase 1
循环计数 ≥ 3？→ 质疑架构，与用户讨论
```

**循环计数规则**：
- 每次修复尝试后必须走完回归验证
- 回归验证失败时递增计数
- 达到 3 次时必须停止并讨论

## 红旗 - 停止并遵循流程

如果你发现自己在想：
- "暂时快速修复，以后调查"
- "试着改 X 看看是否有效"
- "添加多个改动，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说 X 但我会不同地适应"
- "这是主要问题：[列出没有调查的修复]"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（当已经尝试 2+ 次时）**
- **每次修复在不同地方揭示新问题**

**所有这些都意味着：停止。回到 Phase 1。**

**如果 3+ 次修复失败：** 质疑架构（见 Phase 4 Step 10）

---

## 用户信号表明你做错了

**注意这些重定向：**
- "那没有发生吗？" - 你假设了但没有验证
- "它会显示...吗？" - 你应该添加证据收集
- "停止猜测" - 你在没有理解的情况下提出修复
- "深度思考这个" - 质疑基础，不只是症状
- "我们卡住了吗？"（沮丧）- 你的方法不工作

**当你看到这些：停止。回到 Phase 1。**

## 常见合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 Bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测-检查-折腾更快。 |
| "先试这个，然后调查" | 第一次修复设定模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未测试的修复不会持久。先写测试证明它。 |
| "一次多个修复节省时间" | 无法隔离什么有效。制造新 Bug。 |
| "参考太长，我会适应模式" | 部分理解保证 Bug。完整阅读。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修复。 |

```markdown

created: <ISO 8601 UTC>
issue: <一句话描述问题>
status: investigating | fixed | escalated
---

# 调试会话：<主题>

## 症状
[观察到的问题]

### Phase 1：证据收集
- 错误信息：[完整错误]
- 复现步骤：[步骤]

```

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **0. 查知识库** | 搜索历史 Bug、评估匹配、决策分支 | 找到匹配或确认无匹配 |
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找工作的例子、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决、测试通过 |
| **4.7. 批量修复** | 提取模式、全局搜索、逐个修复 | 所有同类 Bug 已修复 |
| **4.8. 存知识库** | 创建知识卡、关联同类 Bug | 知识卡完整可复用 |

---

## 当流程揭示"没有根因"

如果系统化调查揭示问题确实是环境的、时间依赖的或外部的：

1. 你已完成流程
2. 记录你调查了什么
3. 实施适当的处理（重试、超时、错误消息）
4. 为未来调查添加监控/日志

**但是：** 95% 的"没有根因"案例是不完整的调查。

## 支持技术

- **根因追踪** - 向后追踪 Bug 通过调用栈找到原始触发
- **深度防御** - 在找到根因后在多层添加验证
- **基于条件的等待** - 用条件轮询替代任意超时
