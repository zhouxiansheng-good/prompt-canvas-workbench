## A组：声明时门禁（写代码前执行）

### 1. 依赖验证门禁

**触发条件**：声明意图中包含"安装依赖"/"引入包"/"import新模块"

**执行动作**：

1. 读取 `safeguards/dependency.md`
2. 按3步验证流程执行（存在性→版本→安全）
3. 验证通过 → 继续安装
4. 验证失败 → 停止安装，按safeguard中的失败处理流程执行
5. 记录验证结果到变更报告

### 2. 安全编码门禁

**触发条件**：声明意图或代码中包含安全触发关键词（auth/login/password/token/SQL/exec/upload等，完整清单见 `safeguards/security.md`）

**执行动作**：

1. 读取 `safeguards/security.md`
2. 识别安全类别（注入/认证/泄露/DevOps）
3. 对照漏洞模式清单逐项检查
4. 发现🔴严重漏洞 → 必须修复后才能继续
5. 记录安全检查结果到变更报告

### 3. 跨文件重构门禁

**触发条件**：声明意图中包含"重构"/"重命名"/"移动文件"/"修改接口"/"提取函数"

**执行动作**：

1. 读取 `safeguards/refactor.md`
2. 执行重构前影响分析（识别变更点→调用方→依赖方→影响矩阵）
3. 检查循环依赖风险
4. 检查接口契约兼容性
5. 按重构范围选择执行策略，重构后跑全量回归测试

### 4. 专项场景门禁

**触发条件**：检测到特定场景关键词

| 触发关键词 | 加载文件 | 检查重点 |
|-----------|----------|----------|
| SQL/query/database/ORM | `safeguards/database.md` | N+1查询/索引/事务/分页 |
| async/thread/lock/concurrent | `safeguards/concurrency.md` | 竞态条件/死锁/资源泄漏 |
| microservice/service mesh/API gateway | `safeguards/microservice.md` | 服务边界/熔断器/分布式追踪 |
| legacy/migrate/undocumented | `safeguards/legacy.md` | 隐式契约/最小改动/先加测试 |
| 中文/编码/乱码/镜像/国内 | `safeguards/chinese-dev.md` | 中文编码/文档适配/生态差异 |

### 5. 变更前影响分析

**触发条件**：修改已有代码前必须执行

**执行流程**：

```
BEFORE 修改任何已有代码：

1. 定位变更点：明确要修改的函数/类/模块
2. 识别调用方：谁在调用这个函数？改了会影响谁？
3. 识别依赖方：这个函数依赖谁？改了会不会破坏依赖链？
4. 构建影响矩阵：
   | 调用方 | 文件 | 行号 | 影响类型 | 风险等级 |
5. 确认安全：影响范围内有无高风险调用方？
```

**示例**：

```text
我要修改：parseConfig() 函数，增加 strictMode 参数
因为：Bug #42 - 配置解析不校验类型

影响分析：
| 调用方 | 文件 | 行号 | 影响类型 | 风险 |
| initApp() | src/app.ts:15 | 直接调用 | 中 - 需传参 |
| loadUserConfig() | src/config.ts:42 | 直接调用 | 低 - 有默认值 |

安全确认：所有调用方都有默认值保护，可以安全修改
修改方案：strictMode 默认 false，保持向后兼容
```

### 9. 前端设计方向确认门禁

**触发条件**：声明意图中包含前端/UI任务关键词（Web组件/页面开发/UI/UX/样式化/前端/landing/仪表盘/表单美化等）

**执行动作**：

1. 读取 `frontend-design/design-thinking.md`
2. 执行设计思考5步流程（目的分析→调性选择→约束确认→差异化定义→设计决策文档）
3. 确认美学方向已选定（11种美学方向中明确选择一种）
4. 确认设计决策文档（design-decision.md）已创建
5. 确认设计约束（字体/色彩/布局）已明确
6. 未完成设计思考 → 先完成设计思考，再继续声明意图

### 10. 美学合规检查门禁

**触发条件**：前端/UI任务的声明意图阶段自动触发

**执行动作**：

1. 读取 `frontend-design/aesthetic-checklist.md`
2. 逐项检查7项美学合规（字体合规/色彩合规/布局合规/动效合规/背景合规/差异化合规/一致性合规）
3. 读取 `frontend-design/banned-patterns.md`，确认未使用禁止模式
4. 发现禁止模式 → 必须替换后才能继续
5. 美学合规检查结果记录到变更报告

## B组：完成时门禁（代码变更完成后执行）

### 6. 文档同步门禁

> 解决P-062文档漂移/P-065只写代码不写文档/P-066重复代码/P-068变动激增

**触发条件**：代码变更完成后自动触发

**执行动作**：

1. 读取 `safeguards/maintenance.md`
2. 对照文档同步检查清单，确定需要同步的文档
3. 检查文档是否已同步更新（未同步→提醒同步，无文档→创建文档）
4. 检测重复代码（相似度>70%→提醒抽取公共函数）
5. 变动管控：单次变更>10个文件或>500行→建议拆分提交
6. 架构决策必须记录ADR
7. 隐性知识捕获：业务规则/设计意图/已知限制/安全假设必须标注

**关键原则**：代码改了文档没改 = 变更未完成

### 7. 回归防护门禁

> 解决P-070回归错误频发/P-071权宜之计补丁积累

**触发条件**：修改已有代码时自动触发

**执行动作**：

1. 读取 `safeguards/team.md`
2. 修改前：运行全量测试，记录基线（通过数/失败数）
3. 修改后：运行全量测试，对比基线（新增失败→回归！必须修复）
4. 检测权宜之计补丁（TODO/FIXME/HACK/硬编码/临时绕过），记录技术债务
5. 补丁管控：每个HACK必须有对应Issue，补丁存活不超过2周，同一问题不许打3次补丁

**关键原则**：回归测试新增失败 = 变更未完成；同一问题3次补丁 = 需要架构重构

### 8. 认知防护门禁

> 解决P-055调试能力退化/P-056技能萎缩

**触发条件**：进入系统化调试时自动触发

**执行动作**：

1. 读取 `safeguards/skill-preserve.md`
2. 调试时遵循"引导而非替代"原则：
   - 先让开发者尝试定位Bug（给3分钟思考时间）
   - 开发者给出假设 → AI验证假设，提供线索，不直接给答案
   - 开发者卡住 → AI逐步引导，每步让开发者先尝试
3. Bug修复后，开发者复述根因（能复述=理解到位，不能=需重新理解）
4. 技能萎缩防护：连续5次完全依赖AI生成时，提醒开发者自己写一个简单功能

### 11. 设计验证门禁

**触发条件**：前端/UI任务代码变更完成时自动触发

**执行动作**：

1. 读取 `frontend-design/banned-patterns.md`
2. 扫描代码中是否存在禁止模式（禁止字体/禁止色彩/禁止布局/禁止设计模式/禁止动效）
3. 读取 `frontend-design/aesthetic-checklist.md`，执行美学合规7项检查
4. 对照设计决策文档（design-decision.md），验证实现是否符合设计方向
5. 发现禁止模式 → 必须替换；发现美学不合规 → 修复后重新检查
6. 设计验证结果记录到变更报告

**关键原则**：前端任务禁止模式未清除 = 变更未完成

### 12. 前端E2E验证门禁

> 融合自 Anthropic webapp-testing。前端任务完成时，除单元测试外还需E2E验证。

**触发条件**：前端/UI任务代码变更完成时自动触发

**执行动作**：

1. 判断测试路径（决策树）：
   - 静态 HTML → 读源码识别选择器 → Playwright 验证
   - 动态 WebApp → 用 `webapp-testing/scripts/with_server.py` 启动服务器 → Reconnaissance-Then-Action
2. 读取 `webapp-testing/SKILL.md` 获取完整决策树和最佳实践
3. 执行 Playwright E2E 验证脚本
4. 采集截图证据 + 控制台日志
5. 验证 DoD 中涉及前端交互的条目

**关键原则**：前端任务无E2E验证证据 = 验收未完成
