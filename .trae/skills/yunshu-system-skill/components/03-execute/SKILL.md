# 03-execute：集中优势兵力执行

> **"集中优势兵力，各个歼灭敌人"** — 一次只解决一个核心问题，不分兵。

## 职责

按计划分步执行，TDD 驱动实现，每步留下证据，失败时系统排查而非盲目重试。核心原则：**一次一个核心问题，集中力量打歼灭战**。

## Step 1：加载与审查计划

**目标**：理解计划全貌，发现潜在问题，在动手前解决。

**方法**：

1. 读取计划文件
2. 提取所有任务文本和上下文（一次性提取，后续不再重复读取）
3. 审查计划：有没有疑问或担忧？
   - 有 → 向用户提出，等确认后再开始
   - 无 → 使用当前平台的计划/任务跟踪能力列出任务；无专用工具时维护一份可见任务清单，开始执行
4. 识别高风险任务（涉及删除/数据库/大规模重构），标注需要用户确认
5. **子智能体模式评估**：判断是否应使用子智能体驱动开发（与 02-plan 路由决策一致）
   - 任务 ≥3 个且基本独立？→ 切换到 `components/07-subagent/SKILL.md`（优先推荐）
   - 需要并行加速？→ 切换到 `components/07-subagent/SKILL.md`
   - 用户要求 `/subagent` 或 `/parallel`？→ 切换到 `components/07-subagent/SKILL.md`
   - 任务紧耦合或需调试？→ 留在本组件串行执行
   - 当前平台 adapter 不允许或未授权子智能体？→ 留在本组件串行执行

### 2.2 TDD 循环（红-绿-重构）

**生产代码行为变更铁律**：
```
没有失败测试 → 不写生产代码
```

文档、配置、流程、审计和纯校验任务不强行走 RED-GREEN；必须改用声明级验证、链接检查、结构化校验、编译检查或等价的新鲜证据。

**RED — 写失败测试**：
1. 写一个最小测试，描述期望行为
2. 测试名要清晰描述行为（不要 `test1`）
3. 用真实代码，避免 mock（除非不可避免）

**【前端任务】RED 增强流程**：
1. 先写设计合规测试（验证美学方向：字体/色彩/布局是否符合设计决策文档）
2. 写功能测试（验证行为正确）
3. 写可访问性测试（验证 WCAG 合规）

**验证 RED — 看它失败**：
1. 运行测试
2. 确认：测试失败（不是报错）、失败原因正确（功能缺失，不是拼写错误）
3. 测试通过了？说明测的是已有行为，修正测试

**【前端任务】验证 RED 增强**：
1. 运行设计合规测试，确认失败（设计框架尚未实现）
2. 运行功能测试，确认失败
3. 运行可访问性测试，确认失败

**GREEN — 写最小实现**：
1. 写最简单的代码让测试通过
2. 不要加额外功能、不要顺手重构、不要"顺便改进"
3. YAGNI（你不需要它）：只写测试要求的

**【前端任务】GREEN 增强流程**：
1. 先实现设计框架（CSS 变量定义/基础布局/背景效果）
2. 在设计框架内实现功能代码
3. 保持设计一致性

**验证 GREEN — 看它通过**：
1. 运行测试
2. 确认：测试通过、其他测试仍然通过、输出干净（无错误/警告）
3. 其他测试失败了？立即修复

**REFACTOR — 清理**：
1. 只在 GREEN 之后
2. 去重、改善命名、提取辅助函数
3. 不加新行为
4. 重构后再跑测试，保持 GREEN

**【前端任务】REFACTOR 增强流程**：
1. 检查设计一致性：所有元素使用 CSS 变量（无硬编码颜色/字体）、动效风格统一、间距体系一致、视觉层次清晰
2. 去重、改善命名
3. 不加新行为

**TDD 反模式**：
- 🚫 "太简单不需要测试" — 简单代码也会出 Bug，测试只需 30 秒
- 🚫 "先写代码再补测试" — 后补的测试通过不能证明任何事
- 🚫 "我已经手动验证了" — 手动验证不可复跑、不可记录
- 🚫 "删掉重写太浪费" — 沉没成本谬误，不可信的代码才是真浪费

**TDD 例外**（需记录原因并提供替代证据）：
- 一次性原型
- 生成代码
- 配置文件
- 文档、流程、审计和纯配置校验任务（改用声明级验证、链接检查、结构化校验等证据）

### 2.3 证据化验证

**铁律**：
```
没有新鲜验证证据 → 不许宣称完成
```

**门禁函数**：识别→运行→读取→验证→声明，跳过任何步骤 = 撒谎。

**验证类型**：编译验证 / 单元测试 / 代码检查 / 运行验证 / UI 验证 / 编译错误汇总 / 端到端冒烟 / CLI 交互验证 / UI 交互验证 / CI 检查验证

**硬规则**：不在脑中验证 / 不信任缓存 / 部分通过=失败 / 每个声明单独验证 / 不用"应该""大概"

→ 详见 [verify.md](verify.md)：门禁函数完整流程、声明级科学验证（可证伪声明→基线→处理→对比→三态判决）、验证工具箱（CLI: tmux/PTY，UI: Playwright/CDP）

### 2.4 代码自检

每个任务完成后，快速自检：

| 检查项 | 说明 |
|--------|------|
| 规范符合 | 代码是否实现了计划中描述的功能？有没有多做或少做？ |
| 类型一致 | 后续任务引用的函数名/属性名是否与前面定义的一致？ |
| 占位符扫描 | 有没有 TODO/TBD/待定？有则补齐 |
| 边界情况 | 空输入、错误输入、并发情况是否处理？ |
| AI 冗余 | 有没有多余注释、异常防御性检查、`any` 转换、深度嵌套、风格不一致？ |
| 美学合规 | 【前端任务】是否通过美学合规检查？详见 `frontend-design/aesthetic-checklist.md` |

**美学合规检查**（前端/UI任务时触发）：
1. 读取 `frontend-design/aesthetic-checklist.md`
2. 逐项检查7项（字体合规/色彩合规/布局合规/动效合规/背景合规/差异化合规/一致性合规）
3. 发现问题 → 修复 → 重新检查
4. 全部通过 → 标记美学合规 ✅

**AI缺陷模式深度检查**（代码自检未通过时触发）：

当自检发现≥2个问题，或涉及逻辑漏洞/回归风险时：

1. 读取 `safeguards/code-quality.md`
2. 按其中的4类缺陷模式清单逐项检查（逻辑漏洞→风格混乱→回归Bug→架构缺陷）
3. 按判定标准决定：通过/修复/架构审查
4. 严重问题必须修复后才能标记完成

### 2.5 标记完成 + 证据留底

1. 在当前平台的计划/任务跟踪记录中标记任务完成；无专用工具时更新可见任务清单
2. **同步状态到 `.yunshu/` 上下文账本**（强制，替代 `.plans/STATE.md` 作为默认机制）：
   ```bash
   python scripts/yunshu.py context record \
     --task-id <task_id> \
     --phase execute \
     --source "<修改的文件列表>" \
     --finding "<任务完成后的关键发现>" \
     --action "<本任务已完成动作>" \
     --gap "<仍缺什么>" \
     --next-read "<下一步需读取的文件>"
   ```
   - 如有 `.plans/STATE.md`，同步更新（可选）
   - `.yunshu/context/` 是状态同步的唯一强制目标
3. **记录验证证据到 `.yunshu/`**（强制）：
   ```bash
   python scripts/yunshu.py verify run \
     --claim "任务 <task_id> 完成: <简述>" \
     --command "<可复跑验证命令>"
   ```
3. **更新上下文账本**（强制）：
   ```bash
   python scripts/yunshu.py context record \
     --task-id <task_id> \
     --phase execute \
     --source "<修改的文件列表>" \
     --finding "<任务完成后的关键发现>" \
     --action "<本任务已完成动作>" \
     --gap "<仍缺什么>" \
     --next-read "<下一步需读取的文件>"
   ```
4. 执行输出充分性四问，确认状态更新包含：做了什么、改了哪些文件、验证结果、缺口/限制、下一步
5. 简要说明：做了什么、验证结果、下一步

**硬规则**：没有记录验证证据 → 不许标记任务完成

## Step 3：系统化调试

**铁律**：
```
没有根因调查 → 不许提修复方案
```

**何时使用**：测试失败、生产环境 Bug、意外行为、性能问题、构建失败、集成问题。特别在时间紧迫/"只是快速修复"/已尝试多个修复时使用。

**CI 失败快速入口**：`gh pr checks` 识别失败 → `gh run view --log-failed` 获取日志 → 定位根因 → 最小化修复 → 推送验证

**四阶段流程**：

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| Phase 0 | 查 Bug 知识库 | 找到匹配或确认无匹配 |
| Phase 1 | 根因调查 | 理解什么和为什么 |
| Phase 2 | 模式分析 | 识别差异 |
| Phase 3 | 假设与测试 | 确认或新假设 |
| Phase 4 | 实施修复 | Bug 解决、测试通过 |

**自纠正循环**：验证失败 → Phase 0-4 → 通过则继续，失败则循环+1 → ≥3 次质疑架构

**红旗**：如果你在想"快速修复试试""跳过测试""再试一次修复"→ 停止，回到 Phase 1

→ 详见 [debug.md](debug.md)：Phase 0-4 完整流程、增量验证修复法、深度防御四层模型、回归验证、同类型 Bug 批量修复、Bug 知识库存储、自纠正循环、红旗信号、合理化陷阱、调试会话记录

## Step 4：上下文压缩与阶段门控

### 4.1 退化信号检测（每完成一个任务后自检）

> 对照SKILL.md「会话健康规则」中的退化信号清单执行自检。

| 信号 | 自检问题 | 是/否 |
|------|----------|-------|
| 记忆混乱 | 我是否重复问了已确认的问题？是否自相矛盾？ | |
| 质量下降 | 最近输出的代码是否有低级错误？是否忽略了已知约束？ | |
| 响应变慢 | 思考时间是否明显变长？输出是否冗余增加？ | |
| 上下文遗忘 | 是否需要重新读取之前已读的文件？是否丢失了任务上下文？ | |

**评估结果**：
- 全部否 → 健康，继续
- 1个🟠信号 → 注意，保存上下文包后继续
- 任何🔴信号或≥2个信号 → 警告，保存上下文包并建议换会话

### 4.2 阶段门控检查（阶段过渡时执行）

> 从03-execute过渡到04-accept时必须执行。

**门控流程**：
1. 执行4.1退化信号检测
2. **创建阶段检查点**（强制）：
   ```bash
   python scripts/yunshu.py checkpoint create \
     --task-id <task_id> \
     --phase execute \
     --summary "<执行阶段完成摘要>" \
     --completed "<已完成任务列表>" \
     --pending "<待验收任务列表>" \
     --artifact "<关键产物路径>" \
     --next-action "进入04-accept验收"
   python scripts/yunshu.py validate checkpoint .yunshu/checkpoints/<checkpoint_id>/checkpoint.json
   ```
3. 创建交接文档（使用增强模板，包含上下文包5类信息）
4. 创建阶段记忆卡（使用增强模板，包含退化信号记录和恢复优先级）
5. 评估结果为「警告」→ 建议用户换会话
6. 评估结果为「健康」或「注意」→ 继续下一阶段

### 4.3 压缩触发条件（具体化）

当发现以下信号时，必须执行压缩：
- 🔴 退化信号检测中任何一项为「是」
- 🔴 已完成≥5个任务（长执行链路）
- 🟠 任务链路过长（多阶段、多模块、多轮执行）
- 🟠 准备切换模型/切换对话/重新进入执行
- 🟠 计划中标注了「会话断点」且已到达

### 4.4 上下文效率检查

> 当上下文接近容量上限时，加载 `safeguards/context.md` 执行效率优化。

**触发条件**（满足任一即触发）：
- 本次会话已读取>10个文件
- 本次会话已完成>15轮交互
- 同一文件被重复读取>2次
- 发现注意力稀释信号（忽略早期约束/遗忘已确认决策）

**执行动作**：
1. 读取 `safeguards/context.md`
2. 执行上下文容量评估
3. 按分层加载规则重新组织上下文
4. 清理过时信息和污染源
5. 评估结果为「警告」→ 触发P-003会话健康规则的门控检查

- `checkpoint.json`（包含 task_id / checkpoint_id / 当前阶段 / 已完成 / 未完成 / 关键产物路径）

优先使用 `python scripts/yunshu.py checkpoint create ...` 写入 `.yunshu/checkpoints/<id>/`；恢复时以同目录的 `phase_memory.json` 为机器事实来源，`phase_memory_card.md` 仅作为人工兜底模板。

### 恢复方式

1. 新开对话（或清空当前对话）
2. 用 `#File` 引用 `phase_memory.json` + `checkpoint.json`
3. 对话开头：**"使用云舒系统，从 checkpoint_id=<...> 恢复继续执行"**
