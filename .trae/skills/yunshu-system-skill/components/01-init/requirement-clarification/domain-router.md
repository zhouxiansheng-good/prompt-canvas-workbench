# 领域需求澄清路由系统

> 解决需求澄清流于表面、无法适配不同业务场景的问题。
> 核心思想：**通用框架保证一致性，领域模板保证深度**。

## 架构

```
用户原始需求
    ↓
[领域识别层] ──关键词匹配──→ 领域分类（电商/网页/程序/通用）
    ↓
[通用基础层] ──必问5题──→ 目标/受众/时限/约束/验收
    ↓
[领域深度层] ──加载领域模板──→ 行业专属问题 + 隐性需求挖掘
    ↓
[反问引擎层] ──模糊检测──→ 自动触发追问，挖深需求
    ↓
[任务卡生成层] ──整合全部信息──→ 输出增强任务卡
```

## 领域识别规则

### 关键词映射表

| 领域ID | 领域名称 | 触发关键词 | 优先级 |
|--------|----------|-----------|--------|
| `ecommerce` | 电商 | 电商、店铺、商城、卖货、商品、订单、支付、购物车、SKU、淘宝、京东、拼多多、独立站、Shopify、亚马逊 | 高 |
| `webapp` | 网页/Web应用 | 网站、网页、官网、落地页、H5、前端、页面、UI、设计、展示、博客、门户 | 高 |
| `software` | 程序/软件开发 | 程序、软件、APP、应用、系统、后端、API、接口、数据库、算法、工具、脚本、爬虫、小程序 | 高 |
| `general` | 通用 | 无匹配关键词时默认 | 低 |

### 识别逻辑

```
1. 提取用户原始需求文本
2. 逐词匹配各领域关键词表
3. 统计匹配命中次数
4. 命中次数最高且 ≥1 的领域 → 确定领域
5. 无命中 → 通用领域
6. 多个领域同分 → 调用当前平台 adapter 指定的交互工具让用户选择
```

### 歧义处理

当用户输入同时命中多个领域（如"做一个电商网站"同时命中电商+网页）：

```
命中多个领域时：
  ├─ 电商权重 > 网页权重？（电商是业务类型，网页是交付形式）
  │   ├─ 是 → 选择电商领域（业务类型优先于交付形式）
  │   └─ 否 → 调用当前平台 adapter 指定的交互工具让用户确认
  └─ 规则：业务类型（电商/程序）> 交付形式（网页/APP）
```

## 执行流程

### Step 1：领域识别

**输入**：用户原始需求文本
**输出**：`domain_id` + 置信度

**方法**：
1. 关键词匹配（见上表）
2. 如置信度 < 0.7 → 向用户确认领域选择
3. 记录识别结果到上下文账本

### Step 2：加载通用基础模板

所有领域共用，必问5题：

| # | 问题 | 维度 | 用途 |
|---|------|------|------|
| G1 | 核心目标是什么？ | 目的 | 明确用户要解决的问题 |
| G2 | 面向哪些人群使用？ | 受众 | 影响功能优先级和体验设计 |
| G3 | 预期完成时间？ | 时限 | 影响技术选型和范围控制 |
| G4 | 有无硬性规则/限制？ | 约束 | 技术栈、预算、合规等 |
| G5 | 怎样算完成/达标？ | 验收 | 定义DoD的基础 |

**提问方式**：
- 优先使用当前平台 adapter 指定的交互方式；Trae 才使用 `AskUserQuestion`，Codex / 其他平台按对应 adapter 降级
- 用户可输入"跳过"使用默认值
- 每个问题记录回答到上下文账本

### Step 3：加载领域专属模板

根据 `domain_id` 加载对应模板文件：

```
templates/
  ├── general.md          # 通用领域（无匹配时）
  ├── ecommerce.md        # 电商领域
  ├── webapp.md           # 网页/Web应用领域
  └── software.md         # 程序/软件开发领域
```

**模板结构**：

```markdown
# <领域名称> 需求澄清模板

## 领域专属问题清单

| # | 问题 | 选项示例 | 优先级 | 隐性需求挖掘点 |
|---|------|----------|--------|---------------|
| ... | ... | ... | 必须/重要/可选 | ... |

## 常见问题模式

### 模式1：<典型场景>
- 用户常这么说："..."
- 实际可能想要："..."
- 建议追问："..."

## 领域术语表

| 术语 | 含义 | 澄清要点 |
|------|------|----------|
| ... | ... | ... |

## 验收标准模板

```yaml
done_definition:
  - "<领域专属DoD 1>"
  - "<领域专属DoD 2>"
  - "<领域专属DoD 3>"
```
```

### Step 4：深度追问引擎

**触发条件**：
- 用户回答模糊（如"差不多""随便""你定"）
- 用户回答存在歧义（如"优化"未指明性能/可读性）
- 关键信息缺失（如电商未提及支付方式）

**反问策略**：

| 模糊类型 | 检测规则 | 反问模板 |
|----------|----------|----------|
| 笼统描述 | 回答 < 5个字 或 包含"随便/差不多" | "能具体描述一下你想要的<XXX>吗？比如..." |
| 歧义词 | 包含多义词（优化/改进/提升） | "你提到的'<词>'是指<选项A>还是<选项B>？" |
| 关键缺失 | 领域模板标记为"必须"的问题未回答 | "还没确认<XXX>，这个会影响<YYY>，能补充一下吗？" |
| 范围不清 | 未明确边界 | "这个需求包含<AAA>吗？还是只做<BBB>？" |

**追问深度控制**：
- 每个问题最多追问 2 轮
- 连续追问后仍模糊 → 记录为"已知缺口"，继续后续流程
- 阻塞级缺失（如电商无支付需求）→ 必须补齐

### Step 5：整合生成增强任务卡

整合通用基础 + 领域专属 + 追问结果，生成结构化任务卡：

```yaml
goal: "<一句话目标>"
domain_id: "<识别出的领域>"
domain_context:
  business_type: "<业务类型>"
  target_users: "<目标用户>"
  delivery_form: "<交付形式：网站/APP/小程序/...>"
done_definition:
  - "<通用DoD 1>"
  - "<通用DoD 2>"
  - "<领域专属DoD 1>"
  - "<领域专属DoD 2>"
constraints:
  - "<时间约束>"
  - "<技术约束>"
  - "<领域专属约束>"
risks:
  - "<识别出的风险>"
non_goals:
  - "<明确不在范围内的事>"
gaps:
  - "<已知但未澄清的信息>"
```

## 与 08-domain-guide 的关系

| 维度 | 本系统（需求澄清路由） | 08-domain-guide（领域引导） |
|------|----------------------|---------------------------|
| 触发时机 | 01-init Step 4 需求澄清阶段 | 01-init Step 4 检测到领域关键词时 |
| 核心目标 | 深度澄清需求，生成增强任务卡 | 选择集收窄，生成领域上下文包 |
| 交互方式 | 问答式澄清（开放+选择） | 选项式引导（纯选择） |
| 输出产物 | 增强任务卡（含domain_id） | 领域上下文包（JSON） |
| 适用场景 | 所有需要深度需求澄清的任务 | 垂直行业工程领域（道路/建筑等） |

**集成规则**：
- 本系统优先：如识别到电商/网页/程序领域 → 使用本系统的深度澄清模板
- 08-domain-guide 兜底：如识别到道路/建筑等传统工程领域 → 进入 08-domain-guide
- 两者可串联：本系统澄清完成后，如需要进一步领域细分（如电商→跨境电商），可再进入 08-domain-guide

## 扩展机制

新增领域模板步骤：

1. 在 `domain-router.md` 关键词映射表新增领域行
2. 在 `templates/` 创建 `<domain_id>.md` 模板文件
3. 遵循模板结构规范（专属问题清单 + 常见模式 + 术语表 + 验收模板）
4. 无需修改主流程代码

## 留底要求

每个阶段必须记录上下文账本：

```bash
# 领域识别后
python scripts/yunshu.py context record \
  --task-id <task_id> \
  --phase init \
  --source "domain-router.md, <领域模板文件>" \
  --finding "<领域识别结果：domain_id, 置信度>" \
  --action "<完成的澄清步骤>" \
  --gap "<已知缺口>" \
  --next-read "<计划阶段需读取的文件>"
```
