# 01-init：调查研究与任务卡

> **"没有调查，没有发言权"** — 不读代码库就不许提方案。

## 职责

先调查代码库，再帮用户理清思路，然后澄清需求，把用户需求压缩成**可检查、可执行、可回滚**的任务卡。

<硬门禁>
在用户批准设计之前，不允许进入任何实现阶段、写任何代码、创建任何项目脚手架。无论任务看起来多简单都必须经过澄清。澄清之前必须先调查代码库。
</硬门禁>

<阶段防跳门禁>
01-init 完成前，唯一允许的下一阶段是 02-plan。进入 02-plan 前必须运行：

```bash
python scripts/yunshu.py gate transition \
  --task-id <task_id> \
  --from-phase init \
  --to-phase plan \
  --user-aligned \
  --dod-confirmed \
  --task-card <任务卡或任务账本路径>
```

门禁失败时必须停在 01-init，补调查、补澄清、补任务卡或补用户确认；不得因为用户说“继续/直接做/很简单”而跳到执行阶段。
</阶段防跳门禁>

## 反模式："这太简单了不需要澄清"

每个任务都要走澄清流程。一个 todo 列表、一个单函数工具、一个配置变更——全部都要。"简单"任务恰恰是未审视的假设造成最多返工的地方。澄清可以很短（真正简单的任务几句话就够），但必须做。

## 交互规则

**所有需要用户选择或确认的地方，必须使用当前平台 adapter 指定的交互方式。**

**禁止行为**：
- 在回复文字中列出选项等用户打字回复
- 用 markdown 列表展示选项
- 写"请回复 A/B/C"之类的文字
- 在非 Trae 环境假装调用 `AskUserQuestion`

**正确行为**：
- Trae IDE：优先调用 `AskUserQuestion`
- Codex / 其他：使用平台原生确认机制；没有选择工具时只问一个必要问题
- 问题明确，选项 2-4 个
- 用户选择后立即继续流程

## 流程

### Step -1：项目留底初始化检查（强制）

> **硬规则**：进入任何云舒流程前，必须确认当前项目已初始化 `.yunshu/` 留底系统。

**执行动作**：

```bash
# 1. 检查 .yunshu/ 是否存在
ls .yunshu/ 2>/dev/null || echo "NEED_INIT"

# 2. 如不存在，立即初始化
python scripts/yunshu.py init

# 3. 创建项目级上下文账本（首次）
python scripts/yunshu.py context record \
  --task-id "project-init" \
  --phase init \
  --source "$(pwd)" \
  --finding "<技术栈、项目结构摘要>" \
  --action "初始化 .yunshu/ 目录" \
  --gap "<待调查项>" \
  --next-read "<下次需补读的文件>"

# 4. 验证
python scripts/yunshu.py validate context .yunshu/context/project-init.json
```

**判定**：
- `.yunshu/` 存在且有效 → 继续 Step 0
- 初始化失败 → 记录错误，手动创建目录后继续

### Step 0：会话上下文加载（恢复场景）

> 如果是从上一会话恢复，必须先加载上下文再开始工作。

**触发条件**：用户说"恢复"/"继续"/"从checkpoint恢复"或提供了checkpoint_id

**方法**：

1. 优先运行 `python scripts/yunshu.py checkpoint list` 查看 `.yunshu/checkpoints/`
2. 用户提供 checkpoint_id 时运行 `python scripts/yunshu.py checkpoint resume <checkpoint_id>`
3. 读取 `.yunshu/checkpoints/<id>/phase_memory.json`；它是机器恢复的唯一事实来源
4. 再按需读取同目录 `checkpoint.json` 与 `handoff.md`，补充决策/进展/代码状态/待办/失败尝试
5. 仅在没有 `.yunshu/checkpoints/` 时，才扫描 `docs/handoffs/` 或 `phase_memory_card.md` 作为人工归档兜底
6. 验证现场一致性（git分支、文件是否存在）
7. 向用户确认恢复点后继续

**无检查点或交接文档 → 跳过此步骤，进入 Step 1**

**上下文效率规则**（恢复场景或新会话均适用）：

从上一会话恢复时，上下文空间有限，必须高效利用：

1. 读取 `safeguards/context.md` 中的分层加载规则
2. 按L0→L1→L2→L3优先级加载，不要一次性读取所有文件
3. 恢复时只加载L0（任务卡+目标）和L1（当前任务相关文件）
4. L2/L3按需加载，不要预读

### Step 0.5：可复现上下文预加载

> 解决“澄清阶段每次都从头读”的问题：先查账本，再补调查。

**触发条件**：任何新任务、继续优化、恢复执行或用户要求“接着上次做”。

**方法**：

1. 优先运行 `python scripts/yunshu.py context preload --task-id <task_id> --limit 3`。
2. 任务编号不明确时运行 `python scripts/yunshu.py context preload --query "<用户请求关键词>" --limit 3`，只选最近且相关的账本。
3. `preload` 返回 fresh 时，复用输出中的 findings/gaps/next_read，不再重读全量历史。
4. `preload` 返回 stale/missing 时，仅重读输出列出的刷新来源和 `next_read`。
5. 仅重读以下内容：
   - `stale` 或 `missing` 的来源
   - 账本里标注的 `next_read`
   - 本轮用户新增需求涉及但账本未覆盖的文件/文档
6. 如果没有 context 账本，进入 Step 1 做首次调查，并在阶段结束前补写账本

**硬规则**：账本是“调查结论索引”，不是事实替代品。账本 fresh 且覆盖当前问题时可复用；账本过期、缺字段或与现场冲突时必须补读原始来源。

**上下文读取上限**：默认只预加载 1-3 条最新且相关 context，最多 5 条；不要扫描 `.yunshu/context/`、`.yunshu/checkpoints/` 下所有历史文件。

### Step 1：调查研究（代码库调查）

> **"没有调查，没有发言权"** — 先读代码，再提问。

**目标**：理解当前项目状态和代码库全貌，避免在信息不足的情况下提问。

**方法**：

| 动作 | 具体操作 | 工具 |
|------|----------|------|
| 检查项目结构 | 读取根目录、关键配置文件 | 平台文件读取/列目录工具 |
| 识别技术栈 | 从 package.json / Cargo.toml / pyproject.toml 等推断 | 平台文件读取工具 |
| 定位相关代码 | 搜索与任务相关的文件和模块 | 平台搜索工具（优先 rg/grep 等快速文本搜索） |
| 识别已有模式 | 找到可复用的代码模式 | 平台搜索工具 |
| 理清依赖关系 | 读取 import/require，构建依赖图 | 平台文件读取工具 |
| 检查 GSD 上下文 | 有 `.plans/` 则读取 STATE.md 和 active/PLAN-*.md | 平台文件读取工具 |
| 检查最近变更 | 查看最近的 git 提交或文件修改 | git / 平台搜索或读取工具 |

**产出**：代码库调查报告（内部使用，包含技术栈、相关文件、可复用模式、依赖关系）

**复现留痕**：调查完成后必须运行 `python scripts/yunshu.py context record ...`，至少记录：
- `--source`：本阶段读过的关键文件/目录/外部链接
- `--finding`：压缩后的关键发现
- `--action`：本阶段已完成动作
- `--gap`：仍缺什么信息
- `--next-read`：下一轮只需要补读什么

```bash
python scripts/yunshu.py context record \
  --task-id <task_id> \
  --phase init \
  --source "<本阶段读过的关键文件/目录/外部链接>" \
  --finding "<压缩后的关键发现>" \
  --action "<本阶段已完成动作>" \
  --gap "<仍缺什么信息>" \
  --next-read "<下一轮只需要补读什么>"
python scripts/yunshu.py validate context .yunshu/context/<context_id>.json
```

**硬规则**：没有运行 `context record` → 不许进入 Step 2

### Step 2：范围评估

**目标**：判断任务是否需要拆分为多个子项目。

**方法**：检查用户请求是否描述了多个独立子系统。

**判断标准**：
- 如果请求包含"和"连接的多个独立功能（如"做一个带聊天、文件存储、计费和分析的平台"），立即标记
- 每个子项目应该能独立产出可工作、可测试的软件
- 子项目之间有明确的依赖顺序

**处理方式**：
- 需要拆分 → 调用平台交互工具，问题"这个请求包含多个独立功能，建议拆分为X个子项目，是否同意？"，选项"同意拆分"/"不需要拆分，继续作为一个项目"
- 不需要拆分 → 继续下一步

### Step 3：思路梳理

> **"先理清思路，再明确需求"** — 用户很多时候自己还没想清楚要什么，直接问"你想解决什么问题"是无效的。

**目标**：帮用户从混沌状态进入清晰状态，确保在提问澄清之前，用户已经想清楚自己要什么。

**核心方法**：复述→追问→具象化→确认

**3.1 复述理解**

把用户的原始需求用自己的话复述一遍，让用户确认理解是否正确。

调用平台交互工具：
- 问题："我理解你的需求是：[复述内容]，这个理解对吗？"
- 选项："对，就是这个意思" / "不完全对，我补充一下" / "完全不对，我重新说"

**3.2 追问本质**

通过追问找到需求背后的真实动机。

调用平台交互工具：
- 问题："你想要这个功能的根本原因是？"
- 选项："解决现有痛点" / "抓住新机会" / "规避风险" / "其他原因"

如果用户选"解决现有痛点"，继续追问："具体是什么痛点？"
如果用户选"抓住新机会"，继续追问："具体是什么机会？"
如果用户选"规避风险"，继续追问："具体是什么风险？"

**3.3 具象化场景**

把抽象需求变成具体场景，让用户能看到画面。

调用平台交互工具：
- 问题："如果这个功能做好了，你会在什么场景下使用？"
- 选项："日常开发时（高频使用）" / "特定场景偶尔使用（低频但关键）" / "演示/展示用" / "其他场景"

**3.4 确认思路清晰度**

调用平台交互工具：
- 问题："现在你的思路清晰了吗？"
- 选项："清晰了，可以继续明确需求细节" / "还不够清晰，需要继续讨论" / "我需要换个角度重新想"

**思路不清晰时的处理**：
- 用户选"继续讨论" → 回到 3.2，换个角度追问
- 用户选"换个角度" → 从不同维度重新梳理（技术视角/用户视角/业务视角）
- 最多循环 3 次，3 次后仍不清晰 → 建议用户先想清楚再继续，不要硬推

**关键原则**：
- 不在用户思路不清时推进到需求澄清
- 不替用户做决定
- 帮用户发现自己的真实需求
- 用具体场景代替抽象描述

### Step 3.5：业务背景挖掘门禁

> 解决P-015业务理解缺失：AI不能只接受字面指令，必须主动理解业务背后的"为什么"。

**触发条件**：用户提出新任务/新功能时自动触发

**执行动作**：

1. 读取 `safeguards/understanding.md`
2. 从8个维度中主动追问至少3个（用户是谁/业务目标/成功指标/现有流程/边界条件/关联系统/数据流向/非功能需求）
3. 评估理解深度：
   - 🟢 覆盖≥5个维度 → 可以进入Step 4
   - 🟡 覆盖3-4个维度 → 补充追问1-2轮
   - 🔴 覆盖<3个维度 → 必须继续追问，不能跳过
4. 如用户提供了需求文档，按「需求文档自检清单」检查文档质量

**关键原则**：理解深度不足时，不许进入Step 4逐个提问澄清

### Step 3.5b：元认知与输出充分性门禁

> 解决P-015业务理解缺失/P-053理解债/P-054主观偏差：在输出任何内容前，强制自检信息完整性和用户可行动性。

**触发条件**：Step 3.5业务背景挖掘完成后、进入Step 4之前自动触发

**执行动作**：

1. 读取 `safeguards/meta-cognition.md`
2. 执行五维信息审视（重点维度1意图 + 维度2约束）
3. 执行输出充分性四问：是否能产出目标/DoD/约束/非目标，是否答全用户显式问题
4. 标记所有可查证缺口、🟡非阻塞缺口、🔴阻塞缺口
5. 可查证缺口 → 先查文档/代码/历史上下文；🔴阻塞缺口 → 只问一个必要问题；🟡非阻塞缺口 → 明示假设后继续
6. 无阻塞缺口 → 进入Step 4

**自检示例**：
```
【元认知自检】
维度1（意图）：用户要"优化代码"——"优化"是歧义词，未确认是性能还是可读性 → 🔴
维度2（约束）：用户未提及时间约束和技术栈约束 → 🔴
维度3（上下文）：已读取相关文件 → ✅
维度4（假设）：假设用户要性能优化——无证据支持 → 🔴
维度5（歧义）："优化"=歧义，"尽快"=无时间标准 → 🔴
→ 4个🔴缺口，必须先向用户确认
```

**关键原则**：有🔴阻塞缺口时不许进入Step 4，不许把猜测包装成已确认信息

### Step 3.6：专项评估门禁

> 解决P-038中文编码/P-054主观偏差/P-075入门路径：根据项目特征按需加载专项防护。

**触发条件**：Step 3.5业务背景挖掘完成后自动评估

| 评估维度 | 触发条件 | 加载文件 | 检查重点 |
|----------|----------|----------|----------|
| 中文开发者适配 | 目标用户包含中文用户/团队在国内 | `safeguards/chinese-dev.md` | 中文编码规范/文档适配/生态差异 |
| 认知偏差校准 | 开发者首次使用AI辅助/对AI能力有误解 | `safeguards/bias-calibration.md` | 主观偏差校准/监督者角色防护 |
| 人才梯队保护 | 团队有初级开发者/任务可分级 | `safeguards/talent-pipeline.md` | 入门路径保护/知识传承机制 |

**执行方式**：按触发条件逐项评估，满足条件则加载对应safeguard执行检查，不满足则跳过

### Step 4：需求澄清（增强版：通用框架 + 领域差异化模板）

**目标**：将模糊的用户意图转化为可执行的任务描述，通过领域差异化模板实现深度澄清。

**核心架构**：通用基础提问 → 领域识别 → 领域专属深度提问 → 反问引擎 → 增强任务卡

**执行流程**：

#### 4.1 领域识别

**方法**：
1. 提取用户原始需求文本
2. 逐词匹配领域关键词表（见 `requirement-clarification/domain-router.md`）
3. 统计匹配命中次数
4. 命中最高且 ≥1 的领域 → 确定 `domain_id`
5. 无命中 → `general`（通用领域）
6. 多领域同分 → 调用当前平台 adapter 指定的交互工具让用户选择

**歧义处理规则**：业务类型（电商/程序）> 交付形式（网页/APP）

**记录**：
```bash
python scripts/yunshu.py context record \
  --task-id <task_id> \
  --phase init \
  --source "requirement-clarification/domain-router.md" \
  --finding "<领域识别结果：domain_id, 置信度>" \
  --action "完成领域识别" \
  --gap "<如识别模糊，记录待确认项>" \
  --next-read "<下一步加载的领域模板>"
```

#### 4.2 通用基础提问（所有领域必问）

| # | 问题 | 维度 | 提问方式 |
|---|------|------|----------|
| G1 | 核心目标是什么？ | 目的 | 开放提问 |
| G2 | 面向哪些人群使用？ | 受众 | 开放提问 |
| G3 | 预期完成时间？ | 时限 | 选项：A) 1周内 B) 1月内 C) 3月内 D) 无硬性要求 |
| G4 | 有无硬性规则/限制？ | 约束 | 开放提问（预算/技术栈/合规） |
| G5 | 怎样算完成/达标？ | 验收 | 开放提问，引导用户说出3+条标准 |

**规则**：
- 优先使用当前平台 adapter 指定的交互方式；Trae 才使用 `AskUserQuestion`，Codex / 其他平台按对应 adapter 降级
- 用户可输入"跳过"使用默认值
- 每个回答记录到上下文账本

#### 4.3 加载领域专属模板

根据 `domain_id` 加载对应模板：

```
requirement-clarification/templates/
  ├── general.md      # 通用领域（默认）
  ├── ecommerce.md    # 电商领域
  ├── webapp.md       # 网页/Web应用领域
  └── software.md     # 程序/软件开发领域
```

**执行方式**：
1. 读取对应模板文件
2. 按优先级排序提问（必须 → 重要 → 可选）
3. 优先使用当前平台 adapter 指定的交互方式提供选项
4. 记录每个回答

#### 4.4 深度追问引擎

**触发条件**：
- 回答模糊（< 5个字 或 含"随便/差不多"）
- 回答歧义（含多义词：优化/改进/提升）
- 关键信息缺失（模板标记"必须"未回答）
- 范围不清（未明确边界）

**反问策略**：

| 模糊类型 | 检测规则 | 反问模板 |
|----------|----------|----------|
| 笼统描述 | 回答 < 5个字 或 含"随便/差不多" | "能具体描述一下你想要的<XXX>吗？比如..." |
| 歧义词 | 含多义词 | "你提到的'<词>'是指<选项A>还是<选项B>？" |
| 关键缺失 | "必须"问题未回答 | "还没确认<XXX>，这个会影响<YYY>，能补充一下吗？" |
| 范围不清 | 未明确边界 | "这个需求包含<AAA>吗？还是只做<BBB>？" |

**深度控制**：
- 每个问题最多追问 2 轮
- 连续追问仍模糊 → 记录为"已知缺口"，继续
- 阻塞级缺失 → 必须补齐

#### 4.5 整合生成增强任务卡

整合通用基础 + 领域专属 + 追问结果：

```yaml
goal: "<一句话目标>"
domain_id: "<识别出的领域>"
domain_context:
  business_type: "<业务类型>"
  target_users: "<目标用户>"
  delivery_form: "<交付形式>"

  - "<通用DoD 1>"
  - "<通用DoD 2>"
  - "<领域专属DoD 1>"
  - "<领域专属DoD 2>"
constraints:
  - "<时间约束>"
  - "<技术约束>"
  - "<领域专属约束>"

  - "<识别出的风险>"

  - "<明确不在范围内的事>"

  - "<已知但未澄清的信息>"
```

**记录最终任务卡**：
```bash
python scripts/yunshu.py context record \
  --task-id <task_id> \
  --phase init \
  --source "requirement-clarification/templates/<domain_id>.md" \
  --finding "<增强任务卡摘要>" \
  --action "完成需求澄清，生成增强任务卡" \
  --gap "<已知缺口>" \
```

#### 4.6 与 08-domain-guide / 09-software-bridge 的集成

| 场景 | 处理方式 |
|------|----------|
| 识别到电商/网页/程序领域 | 使用本系统的深度澄清模板（优先） |
| 识别到道路/建筑/桥梁等传统工程领域 | 进入 `components/08-domain-guide/SKILL.md` |
| 用户提到"部署""软件操作""CLI" | 澄清完成后，执行阶段进入 `components/09-software-bridge/SKILL.md` |
| 本系统澄清后需进一步领域细分 | 可再进入 08-domain-guide 做选择集收窄 |

**常见陷阱**：
- 用户说"简单"→ 追问具体功能，加载对应领域模板深挖
- 用户说"像XX一样"→ 追问具体哪些功能需要，用领域模板逐项确认
- 用户说"你先做"→ 用模板中的验收标准引导用户确认DoD
- 用户说"后面再加"→ 用模板中的"常见问题模式"提醒扩展性风险

### Step 5：提出 2-3 个方案

**目标**：让用户看到不同路径的权衡，而不是只有一个选择。

调用平台交互工具：
- 问题："以下是 2-3 个实现方案，你倾向哪个？"
- 选项：
  - "方案 A（推荐）：[方案名] — [一句话描述]。优势：[核心优势]。代价：[核心代价]"
  - "方案 B：[方案名] — [一句话描述]。优势：[核心优势]。代价：[核心代价]"
  - "方案 C：[方案名] — [一句话描述]。优势：[核心优势]。代价：[核心代价]"

**原则**：
- 先说推荐方案和理由
- 每个方案必须包含代价/风险，不能只说好处
- YAGNI（你不需要它）：从所有方案中砍掉不必要的功能

### Step 6：分段展示设计

**目标**：让用户逐步确认设计，而不是一次性审批一大段文字。

**方法**：按以下维度分段说明，每段调用一次平台交互工具等用户确认：

| 段落 | 内容 | 粒度 |
|------|------|------|
| 架构 | 整体方案、模块划分、数据流 | 2-3 句话（简单）/ 200-300 字（复杂） |
| 组件 | 每个模块的职责和接口 | 每个组件一段 |
| 错误处理 | 失败场景和恢复策略 | 关键场景一段 |
| 测试 | 验证策略 | 一段 |

**每段展示后**：

调用平台交互工具：
- 问题："以上是[架构/组件/错误处理/测试]设计，这部分看起来对吗？"
- 选项："对，继续下一段" / "不对，需要修改"

**用户说不对 → 回到 Step 4 补充澄清**
**用户说对 → 继续下一段**

### Step 7：生成任务卡

**目标**：把澄清结果压缩为结构化任务卡。

**任务卡结构**：

```yaml
goal: "一句话目标"
done_definition:
  - "DoD 1：[可检查的验收条件]"
  - "DoD 2：[可检查的验收条件]"
  - "DoD 3：[可检查的验收条件]"
constraints:
  - "约束 1"
  - "约束 2"
risks:
  - "风险 1"
non_goals:
  - "不在范围内的事 1"
  - "不在范围内的事 2"
```

**DoD 检查清单**：
- [ ] 至少 3 条 DoD
- [ ] 每条 DoD 都是可检查的（能回答"通过/未通过"）
- [ ] 没有模糊词（如"优化"/"改进"/"提升"——必须量化或具体化）
- [ ] non_goals 明确列出了不在范围内的事

### Step 8：任务卡自检

**目标**：确保任务卡没有遗漏和矛盾。

**自检项**：

| 检查项 | 说明 |
|--------|------|
| 占位符扫描 | 有没有"TBD"/"TODO"/"待定"？有则补齐或删除 |
| 内部一致性 | goal 和 done_definition 是否矛盾？constraints 和 non_goals 是否冲突？ |
| 范围检查 | 任务卡是否聚焦到单个可执行计划？还是需要拆分？ |
| 歧义检查 | 有没有可以有两种理解的表述？有则选一种并明确 |

**发现问题 → 当场修复，不需要重新走流程**

### Step 9：用户审批

**目标**：让用户最终确认任务卡。

调用平台交互工具：
- 问题："任务卡已生成。请确认 goal、DoD、约束和非目标是否准确。"
- 选项："确认，进入计划阶段" / "需要修改"

**用户选"需要修改" → 回到 Step 4 补充澄清，更新任务卡**
**用户选"确认" → 进入 Step 10**

**审批后机器门禁**：

用户确认任务卡后，立刻运行 `gate transition`。只有门禁通过，才能进入 02-plan：

```bash
python scripts/yunshu.py gate transition \
  --task-id <task_id> \
  --from-phase init \
  --to-phase plan \
  --user-aligned \
  --dod-confirmed \
  --task-card .yunshu/context/<context_id>.json
```

### Step 10：项目初始化（可选）与状态同步

> **留底系统统一**：`.yunshu/` 是云舒系统的唯一留底目录。`.plans/` 目录（GSD 风格）仅在用户显式要求时创建，不作为默认留底机制。

**默认行为**：
- `.yunshu/` 已初始化（Step -1）→ 继续同步上下文账本
- 无需创建 `.plans/` 目录

**可选行为**（用户显式要求 GSD 风格项目管理时）：

```bash
# 仅在用户要求时创建 .plans/
mkdir -p .plans/active .plans/completed
# 使用 templates/gsd_*.md 创建对应文件
cp templates/gsd_state.md .plans/STATE.md
```

**状态同步**：
- 优先更新 `.yunshu/context/` 账本
- 如有 `.plans/STATE.md`，同步更新（但不作为唯一来源）

## 产物

- 任务卡（包含 goal / done_definition / constraints / risks / non_goals）
- 至少 3 条可检查的 DoD
- 明确的 non_goals（不在范围内的事）

## 路由

- 信息不足 → 继续澄清，不跳过
- 范围过大 → 拆分为子项目，对第一个子项目重新走澄清流程
