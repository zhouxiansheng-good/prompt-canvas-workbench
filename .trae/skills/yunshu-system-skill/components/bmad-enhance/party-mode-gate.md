# Party Mode Gate：多角色讨论门禁

## 目标

把 BMad Party Mode 限制为复杂决策的可选讨论机制，防止它绕过云舒任务卡、ADR、计划和验收门禁。

## 允许触发

满足以下条件之一时，才允许使用多角色讨论：

- 架构决策存在多个高代价选项。
- PRD、架构和实现之间出现冲突，需要产品 / 架构 / 工程视角共同校准。
- 复盘或事后分析需要区分需求、架构、实现、测试责任。
- 用户明确要求 BMad Party Mode 或多角色讨论。

## 禁止触发

- 单文件修改、简单 Bug 修复、快速问答。
- 任务卡还不存在。
- 讨论没有明确要产出的决策。
- 讨论结果无法进入云舒 ADR、plan/spec/tasks 或验收证据。

## 角色边界

| BMad 视角 | 可贡献内容 | 不允许做的事 |
|-----------|------------|--------------|
| Analyst / PM | 问题、用户、价值、范围 | 直接决定技术实现 |
| Architect | 模块、接口、数据流、风险 | 绕过现有代码证据 |
| Developer | 实现可行性、测试、复杂度 | 跳过云舒执行验证 |
| UX / Writer | 体验、文档、表达清晰度 | 替代验收 |

## 输出要求

多角色讨论结束后，必须压缩为以下云舒产物之一：

- ADR：关键架构或产品决策。
- plan/spec/tasks 更新：具体执行计划。
- risk register：风险和应对。
- Story Map：可分派执行包。
- open questions：返回 01-init 澄清的问题。

## 门禁

讨论结果进入执行前必须满足：

- 有明确决策，不只是观点罗列。
- 决策有证据或明确假设。
- 影响范围已记录。
- 需要用户确认的阻塞问题已提出。
- 验证方式已进入 spec 或 tasks。

## 反模式

- 用热闹讨论替代调查研究。
- 多角色互相“同意”但没有 ADR。
- 讨论后直接写代码。
- 把某个角色的话当作最终验收证据。
